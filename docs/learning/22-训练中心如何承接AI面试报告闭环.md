# 22 训练中心如何承接 AI 面试报告闭环

## 1. 为什么报告不是终点

AI 模拟面试系统如果只做到“面试结束后生成报告”，产品价值是不完整的。报告只能告诉用户哪里不好，但用户真正需要的是下一步怎么练、练完以后怎么验证。

所以训练中心的定位不是普通任务列表，而是承接报告结果的专项训练工作台。

完整链路是：

```text
完成模拟面试
-> 生成面试报告
-> 从报告里提取 weakTag
-> 根据 weakTag 生成训练任务
-> 用户在训练中心专项练习
-> 回到面试台再进行一轮模拟面试
```

这个闭环让项目从“AI 问答工具”更接近“AI 面试训练产品”。

## 2. weakTag 如何变成训练任务

weakTag 可以理解为系统对用户薄弱点的结构化标签。

例如用户在一场面试中多次答不上来：

```text
RAG 命中日志怎么设计
Agent 决策日志怎么排查问题
Tool calling 和普通函数调用有什么区别
```

系统可以把这些问题归纳成：

```text
rag_quality
agent_observability
agent_tool_calling
```

训练任务就是围绕这些 weakTag 生成的：

```text
weakTag: agent_tool_calling
title: 工具调用专项训练
description: 用自己的话解释 Agent 为什么需要 tool calling，并结合项目举例说明。
priority: high
masteryScore: 45
```

这样做的好处是：模型生成的复盘内容不再只是自然语言，而是可以被后续系统继续处理的结构化数据。

## 3. 为什么训练中心要按 weakTag 组织

如果训练任务只按时间倒序排列，用户很难知道自己真正要优先练什么。

按 weakTag 组织以后，用户可以看到：

- 哪个薄弱点任务最多。
- 哪个薄弱点优先级最高。
- 哪个薄弱点平均掌握度最低。
- 当前筛选条件下有哪些待训练、训练中、已完成任务。

这就是 Training Center V2 里“薄弱点训练地图”的意义。

它不是装饰组件，而是把报告里的问题变成可执行训练路线。

## 4. 本项目的训练闭环链路

当前项目里的训练闭环可以这样理解：

```text
ReportPage
  展示面试报告、逐题复盘、weakTags

training API
  根据报告生成训练任务

training store
  管理任务列表、状态筛选、weakTag 筛选、统计信息

TrainingPage
  展示训练概览、薄弱点地图、状态筛选、任务列表

InterviewPage
  用户回到面试台，再验证薄弱点是否改善
```

Training Center V2 前端侧重点不是重写后端，而是把已有训练任务组织得更像正式产品：

- 顶部概览：待训练、训练中、已完成、已归档、平均掌握度。
- 薄弱点地图：按 weakTag 聚合任务。
- 状态筛选：按 todo、in_progress、done、archived 查看任务。
- 任务卡片：展示优先级、掌握度、尝试次数、下次复习时间。
- 回到面试台：训练后继续进入下一轮模拟面试。

## 5. 面试时怎么讲

可以这样表达：

```text
我的项目不是只在面试结束后生成一份报告，而是把报告里的 weakTag 继续沉淀成训练任务。

前端训练中心会按 weakTag 聚合任务，展示待训练、训练中、已完成、平均掌握度和优先级。用户可以围绕某个薄弱点持续训练，训练完成后再回到面试台验证提升效果。

从工程角度看，这里体现的是 AI 应用里的闭环设计：模型输出不是终点，而是会被结构化保存，并驱动后续业务流程。
```

如果面试官继续追问技术实现，可以补充：

```text
前端使用 Vue3 + Pinia。Pinia store 负责维护训练任务列表和筛选状态，TrainingPage 作为容器组件组合概览卡片、weakTag 地图、状态筛选和任务列表。展示型组件只接收 props 和触发事件，这样组件边界更清楚，也更容易做 Vitest 单元测试。
```

## 6. 这一阶段还没有做什么

Training Center V2 主要是前端产品化增强，不等于完整训练系统已经到生产级。

当前还没有重点展开：

- 复杂训练计划日历。
- 定时提醒。
- 学习曲线图。
- 掌握度的模型化评分。
- 个性化训练推荐算法。
- LangGraph 根据训练结果动态调整下一场面试流程。

这些可以作为后续阶段继续升级。
