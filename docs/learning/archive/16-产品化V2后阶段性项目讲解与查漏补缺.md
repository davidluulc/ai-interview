## 2. 当前项目一句话介绍

可以这样说：

```text
我做的是一个面向大学生、应届生和求职者的 AI 模拟面试训练系统。系统支持用户维护求职档案，结合简历、岗位 JD、岗位知识库 RAG、题库 RAG 和候选人画像 RAG 生成动态面试问题。Interview Orchestrator Agent 会根据当前回答质量、RAG 命中、历史薄弱点和训练任务决定下一步是深挖、降难度、换话题还是进入复盘。面试结束后系统会生成报告，并把 weakTags 沉淀为可跟踪训练任务，形成面试到训练的闭环。
```

更短一点可以说：

```text
这是一个带三类 RAG、Agent 决策和训练闭环的 AI 模拟面试系统。
```

## 3. 当前完整业务流程

从用户角度：

```text
注册 / 登录
-> 创建求职档案
-> 输入简历、JD、公司要求
-> 选择学习辅导模式或真实面试模式
-> 开始模拟面试
-> 用户逐题回答
-> 系统动态追问
-> 生成面试报告
-> 生成 weakTags 和训练建议
-> 一键生成训练任务
-> 在训练中心持续练习薄弱点
```

从后端角度：

```text
/api/interview/next-question
-> 读取用户身份
-> 读取三类 RAG
-> 记录 RAG 命中日志
-> 构造 Agent State
-> 读取训练任务 candidateTrainingTasks
-> 生成 Agent Decision
-> normalize / fallback / guardrail
-> 调用模型生成下一题
-> 记录 Agent 决策日志
-> 返回前端
```

## 5. 三类 RAG 如何协作

当前三类 RAG 是：

```text
岗位知识库 RAG
题库 RAG
候选人画像 RAG
```

它们的职责不同：

```text
岗位知识库 RAG：
  让问题贴合岗位 JD、技术栈和业务背景。

题库 RAG：
  给模型参考常见题、难度、考察点和参考答案方向。

候选人画像 RAG：
  召回用户历史回答、薄弱点、训练建议和长期画像。
```

面试时可以这样讲：

```text
我没有把所有资料混成一个知识库，而是拆成岗位知识、题库和候选人画像三类 RAG。岗位知识库解决岗位贴合度，题库解决考察点覆盖，候选人画像解决个性化和多轮记忆。这样检索结果更容易解释，也方便在日志中观察每类 RAG 的命中情况。
```

## 6. Agent 当前能力

当前 Agent 已经具备：

```text
Agent State
ToolCalls
Agent Decision
fallback
normalize
guardrail
nodeTrace
coach / interview 模式
重复问题保护
连续弱回答识别
topic shift
weaknessStrategy
trainingTemplateHint
candidateTrainingTasks
selectedTrainingTask
```

可以这样理解：

```text
普通 LLM 调用是输入 prompt，得到回答。
Agent 是先观察状态，再调用工具，再分析回答，再选择动作，最后生成下一题。
```

当前项目里的 Agent 不只是“会问问题”，而是能回答：

```text
为什么继续深挖？
为什么降低难度？
为什么切换话题？
为什么参考这个训练任务？
为什么这轮使用 fallback？
```

## 7. 训练闭环怎么讲

当前训练闭环是：

```text
面试回答
-> 报告复盘
-> weakTags
-> TrainingTask
-> 训练中心
-> 开始 / 完成 / 归档
-> 掌握度变化
-> Agent 下一轮读取 candidateTrainingTasks
```

这让系统从一次性面试工具变成训练产品。

面试表达：

```text
我把报告里的 weakTags 抽象成 TrainingTask，让薄弱点从一次性诊断结果变成可跟踪任务。任务有 priority、status、masteryScore 和 attemptCount。下一轮面试时，Agent 会读取高优先级低掌握度任务，把它作为个性化决策信号，决定是否在 coach 模式下降低难度继续训练。
```

## 8. 管理员后台怎么讲

当前后台是 MVP，只做只读：

```text
用户列表
RAG 文档列表
RAG 日志
Agent 日志
系统配置摘要
```

核心原则：

```text
前端隐藏后台入口只是体验。
真正权限控制在后端 require_admin_user。
```

面试表达：

```text
我给系统加了管理员后台 MVP，后端把后台能力统一收敛到 /api/admin/*，并用 require_admin_user 做权限控制。第一版只做只读列表，避免一开始引入删除、封禁、复杂 RBAC 等高风险操作。后台主要服务于 AI 应用可观测性，比如查看 RAG 命中日志和 Agent 决策日志。
```

## 9. 当前数据库表关系

可以先记这几类：

```text
User
  用户主表。

RefreshToken
  登录刷新 token，属于某个用户。

ApplicationProfile
  求职档案，属于某个用户。

InterviewRecord
  历史面试记录，属于某个用户，也可以关联某个求职档案。

RagDocument
  RAG 文档，属于某个用户。

RagChunk
  文档切片，属于某个文档。

RagRetrievalLog
  RAG 命中日志，属于某个用户。

AgentDecisionLog
  Agent 决策日志，属于某个用户。

TrainingTask
  训练任务，属于某个用户，也可以关联求职档案和历史面试记录。
```

你要重点理解：

```text
user_id 是数据归属。
ForeignKey 是数据库层面的关联。
relationship 是 Python ORM 层面的对象导航。
```

## 11. 当前项目短板清单

当前还不够强的地方：

```text
1. RAG 工程化还可以继续升级。
   例如文档版本、启用/禁用、metadata filter、检索评估面板、case 管理。

2. 上线部署还没有真实落地。
   当前没有 Docker/Nginx/云服务器真实部署。

3. 前端仍然是单页大页面。
   功能越来越多，需要统一信息架构和页面拆分。

4. LangGraph 还只是迁移预留。
   当前没有真实 StateGraph、checkpoint、interrupt。

5. 管理员后台还是只读 MVP。
   还没有文档审核、禁用、删除、配置修改、操作审计。

6. 数据库仍以 SQLite 本地开发为主。
   上线前需要考虑 PostgreSQL/MySQL、迁移和备份。
```

## 12. 下一阶段候选方向

推荐下一阶段做：

```text
RAG 工程化 V3 + 上线前工程化准备
```

也就是把两件事合并规划：

```text
A：上线前工程化 V3
B：RAG 工程化 V3
```

但实现顺序建议是：

```text
先 B，后 A。
```

原因：

```text
你面试的是 AI 应用开发岗，RAG 工程化更贴合岗位。
但最终项目要上线，部署和数据层准备也不能完全不管。
```
