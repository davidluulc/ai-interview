# 12 训练中心前端页面如何拆分

## 1. 为什么要从面试页拆出训练中心

最早的系统主线是：

```text
开始面试 -> 回答问题 -> 生成报告 -> 看复盘
```

这个流程能证明系统“会面试”，但还不像一个长期训练产品。

训练中心要解决的是另一个问题：

```text
用户结束一轮面试后，第二天回来应该练什么？
每个薄弱点练了几次？
掌握度有没有变化？
这个薄弱点是否已经可以归档？
```

所以训练中心不是简单多加一块页面，而是把 `weakTags` 变成可持续跟踪的任务列表。

## 2. 前端拆分思路

当前没有引入 React、Vue、Next.js，而是继续使用原生 HTML、CSS、JavaScript。

核心拆法是：

```text
index.html：提供稳定 DOM 容器
styles.css：负责训练中心布局和响应式样式
app.js：负责状态、接口请求、渲染和交互
tests/*.mjs：用 Node VM 测试前端函数行为
```

这说明前端工程化不一定等于马上换框架。只要状态边界清楚、渲染函数稳定、测试能覆盖核心行为，原生 JS 也可以支撑 MVP 级产品。

## 3. 页面结构

训练中心页面区域在 `index.html` 中使用三个稳定 ID：

```text
trainingRefreshButton
trainingTaskList
trainingTaskDetail
```

它们分别负责：

```text
trainingRefreshButton：刷新训练任务
trainingTaskList：展示任务列表
trainingTaskDetail：展示当前选中任务详情和操作按钮
```

这样设计的好处是列表和详情分离。

列表回答：

```text
我现在有哪些薄弱点任务？
哪个优先级最高？
掌握度是多少？
```

详情回答：

```text
这个任务具体练什么？
当前状态是什么？
能否开始、完成或归档？
```

## 4. 前端状态如何设计

训练中心的状态保存在：

```javascript
session.training = {
  tasks: [],
  selectedTaskId: null,
};
```

这两个字段足够支撑第一版训练中心：

- `tasks`：后端返回的训练任务列表。
- `selectedTaskId`：当前用户点中的任务。

渲染时通过 `selectedTrainingTask()` 找到当前任务，再生成详情区域。

这是一种很基础但很重要的前端状态管理思想：

```text
不要让页面自己记忆业务状态。
应该让状态对象作为唯一事实来源，然后由渲染函数把状态画到页面上。
```

## 5. 接口流转

训练中心当前使用这些接口：

```text
GET  /api/training/tasks
POST /api/training/tasks/{task_id}/start
POST /api/training/tasks/{task_id}/complete
POST /api/training/tasks/{task_id}/archive
POST /api/training/tasks/generate-from-report
```

它们对应的产品动作是：

```text
打开训练中心 -> 拉取任务列表
点击开始训练 -> 状态变成训练中
点击标记完成 -> 掌握度提升，训练次数增加
点击归档 -> 任务从活跃训练中退出
点击一键重练薄弱点 -> 根据报告 weakTags 生成训练任务
```

## 6. 一键重练为什么要生成训练任务

之前的一键重练只做了一件事：

```text
把薄弱点写入下一轮面试上下文，然后直接开始下一轮面试。
```

这更像临时动作。

现在增加了：

```text
generateTrainingTasksFromLatestReport()
```

它会把当前报告发送到：

```text
POST /api/training/tasks/generate-from-report
```

后端再从报告里的 `weakTags` 提取薄弱点，生成或更新 `TrainingTask`。

这样一键重练就从“临时入口”升级成了“训练闭环入口”：

```text
报告 weakTags -> 训练任务 -> 训练中心可见 -> 下一轮可继续练
```

## 7. 为什么登录、退出也要刷新训练中心

训练任务是用户私有数据。

所以登录态变化时，训练中心必须同步变化：

```text
登录成功 -> 拉取当前用户训练任务
退出登录 -> 清空训练任务并显示登录提示
token 失效 -> 清空登录态并刷新训练中心
```

否则会出现一个严重问题：

```text
用户 A 退出后，页面还显示用户 A 的训练任务。
```

这就是前端安全边界和产品体验都不能接受的状态泄漏。

## 8. 测试覆盖了什么

本轮新增或更新了三个前端测试：

```text
tests/frontend_training_center.test.mjs
tests/frontend_training_actions.test.mjs
tests/frontend_training_events.test.mjs
tests/frontend_interview_flow.test.mjs
```

它们分别验证：

```text
训练任务能加载并渲染。
开始、完成、归档能调用正确接口并更新页面。
刷新、任务选择、详情按钮注册了点击事件。
一键重练会先生成训练任务，再进入下一轮 coach 模式复练。
```

这说明我们不是只靠手点页面判断“好像能用”，而是把关键交互写进测试。

## 9. 面试时怎么讲

可以这样表达：

```text
我把面试报告里的 weakTags 抽象成 TrainingTask，让薄弱点从一次性诊断结果变成可持续跟踪的训练任务。前端训练中心采用列表加详情的结构，列表展示优先级、掌握度和状态，详情支持开始、完成和归档。登录态变化时会自动刷新训练任务，避免用户数据串号。一键重练薄弱点时，前端会先调用 generate-from-report 接口，把报告里的 weakTags 持久化为训练任务，然后再启动下一轮 coach 模式复练。这样系统形成了从面试、诊断、任务生成到再次训练的闭环。
```

如果面试官追问“为什么不直接让报告里显示建议就够了”，可以回答：

```text
报告建议只能解决一次复盘，训练任务解决的是长期跟踪。它有状态、优先级、掌握度和训练次数，后续还能被 Agent 读取，用来决定下一轮是否继续追问某个薄弱点。
```
