# 24. 向量库持久化迁移抽象设计

## 1. 当前项目怎么存向量

当前项目仍然用 SQLite 存 embedding：

```text
rag_chunks.embedding_json
rag_chunks.embedding_model
rag_chunks.embedding_status
```

这对学习项目和 MVP 很合适：

- 部署简单；
- 不需要额外启动 Qdrant / Milvus / Elasticsearch；
- 测试稳定；
- 方便你理解完整链路。

但如果数据量变大，SQLite JSON 字段不适合做真正的向量检索。

## 2. 为什么要做 VectorStore 抽象

如果业务代码直接写死：

```text
从 rag_chunks 表读 embedding_json
手动 cosine similarity
```

以后迁移 Qdrant 或 pgvector 时，要改很多业务代码。

更好的方式是先定义一个接口：

```text
VectorStore
  upsert_embedding()
  search()
```

业务代码只依赖接口，不依赖具体向量库。

当前实现：

```text
SQLiteVectorStore
```

未来可以实现：

```text
QdrantVectorStore
PgVectorStore
```

## 3. 本轮实现了什么

新增：

```text
backend_python/vector_store.py
```

包含：

- `VectorStore` Protocol
- `VectorSearchResult`
- `SQLiteVectorStore`
- `parse_embedding()`
- `cosine_similarity()`

`SQLiteVectorStore` 支持：

- `upsert_embedding()`
  - 写入 chunk embedding；
  - 更新 embedding model；
  - 更新 embedding status。
- `search()`
  - 按向量相似度检索；
  - 遵守文档生命周期；
  - 遵守 private / public 权限边界；
  - 支持 metadata filter；
  - 同分时优先当前用户自己的文档。

## 4. 为什么同分优先用户自己的文档

系统里会同时存在：

```text
用户私有知识库
平台公共知识库
```

当两条资料向量分数相同或接近时，当前用户自己的资料通常更贴近当前面试场景。

所以 SQLiteVectorStore 的排序策略是：

```text
先看是不是当前用户自己的文档
再看向量相似度分数
```

这样可以降低公共知识库“盖住”用户私有简历和岗位资料的概率。

## 5. 和 retrieval_service 的关系

本轮把 `retrieve_vector_chunks()` 改成通过 `SQLiteVectorStore` 检索。

也就是说，上层仍然调用：

```text
retrieve_chunks(..., mode="vector")
```

但底层已经从“直接查表”变成：

```text
通过 VectorStore 适配器查向量
```

这样对上层接口无感知，后续迁移向量库时更稳。

## 6. 面试表达模板

你可以这样讲：

```text
当前项目为了降低部署复杂度，embedding 仍然保存在 SQLite 的 embedding_json 字段中。
但我没有让业务代码直接依赖 SQLite 的存储细节，而是抽象了 VectorStore Protocol。
当前实现是 SQLiteVectorStore，支持 upsert_embedding 和 search，并且 search 会统一处理文档生命周期、private/public 权限边界和 metadata filter。
后续如果迁移 Qdrant 或 pgvector，只需要新增 QdrantVectorStore 或 PgVectorStore 实现同样接口，上层 retrieve_chunks 和 RAG 主流程不需要大改。
这是为了让项目从 MVP 具备向生产级向量库迁移的工程基础。
```
