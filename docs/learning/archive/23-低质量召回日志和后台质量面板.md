# 23. 低质量召回日志和后台质量面板

## 1. 为什么要做低质量召回面板

RAG 系统上线后，最难排查的问题通常不是“接口报错”，而是：

```text
接口正常返回
模型也正常回答
但是回答质量不稳定
```

这时就要看 RAG 召回是否出了问题。

低质量召回面板的作用是让管理员快速看到：

- 哪些 query 空召回；
- 哪些 query 命中很弱；
- 哪些召回结果没有进入 prompt；
- 哪个 retriever 经常出问题；
- 后续应该补知识库、调 query rewrite，还是调整 filter。

## 2. 本轮低质量类型

当前第一版识别三类：

```text
empty_recall      : 空召回，hitCount = 0
weak_recall       : 有召回，但质量等级为 weak
unused_in_prompt  : 有召回，但没有进入 prompt
```

后续可以扩展：

```text
metadata_miss
keyword_coverage_low
rerank_drop_abnormal
duplicate_chunk_pollution
```

## 3. 后端接口

本轮新增：

```text
GET /api/admin/rag/quality
```

只有管理员可以访问。

返回结构：

```json
{
  "summary": {
    "totalLogCount": 80,
    "lowQualityCount": 12,
    "emptyRecallCount": 3,
    "weakRecallCount": 7,
    "unusedInPromptCount": 2
  },
  "items": [
    {
      "queryText": "empty recall query",
      "retrieverName": "role_knowledge",
      "issueType": "empty_recall",
      "recommendation": "补充知识库内容，检查 query rewrite、metadata filter 和知识库类型是否匹配。"
    }
  ]
}
```

## 4. 前端后台面板

管理员后台新增“低质量召回”只读区。

当前展示：

- 低质量总数；
- 空召回数量；
- 弱召回数量；
- 未进入 Prompt 数量；
- 最近低质量召回样例；
- 每条样例的修复建议。

本阶段不做删除、归档、批量修复，避免后台误操作。

## 6. 面试表达模板

你可以这样讲：

```text
我给 RAG 做了可观测性后台，不只是记录日志。
管理员可以通过低质量召回面板查看最近的空召回、弱召回和未进入 prompt 的召回记录。
后端会基于 RagRetrievalLog 的 hitCount、quality.level 和 usedInPrompt 识别问题类型，并给出 recommendation。
前端后台只读展示摘要和样例，第一版不做删除，避免误操作。
这个面板的价值是让 RAG 调优有证据，比如判断应该补知识库、优化 query rewrite，还是调整 metadata filter。
```
