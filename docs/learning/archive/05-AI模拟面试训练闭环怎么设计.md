# 05 AI 模拟面试训练闭环怎么设计

## 1. 什么是训练闭环

训练闭环不是“问完几道题，生成一个分数”就结束。

真正有价值的 AI 模拟面试系统应该形成这样的闭环：

```text
输入简历和岗位 JD
-> 生成面试问题
-> 用户作答
-> 逐题复盘
-> 提取薄弱点
-> 生成下一轮训练计划
-> 用户按计划重练
-> 候选人画像更新
-> 下一轮问题更贴合弱点
```

这条链路能让产品从“模拟面试玩具”变成“持续训练工具”。

## 2. 当前项目已经具备哪些能力

当前项目已经有训练闭环雏形。

### 2.1 面试问题生成

接口：

```text
POST /api/interview/next-question
```

它会结合：

- 用户 profile。
- 历史问答 history。
- 岗位知识库 RAG。
- 题库 RAG。
- 候选人画像 RAG。
- Agent Decision。

生成下一道问题。

### 2.2 面试报告生成

接口：

```text
POST /api/interview/report
```

报告会返回：

- `score`
- `strengths`
- `risks`
- `actions`
- `questionReviews`
- `trainingPlan`

### 2.3 逐题复盘 questionReviews

`questionReviews` 会按每道题生成复盘。

每条复盘包括：

- `index`：第几题。
- `focus`：考察点。
- `question`：题目。
- `answerStatus`：回答状态。
- `whyAsked`：为什么问这题。
- `missingPoints`：缺失点。
- `referenceDirection`：参考回答方向。
- `trainingAction`：下一步训练动作。

这部分很重要，因为它能解释：

```text
不是只给总分，而是能告诉用户每一道题为什么没答好。
```

### 2.4 下一轮训练计划 trainingPlan

`trainingPlan` 会把本轮复盘转成下一轮训练计划。

字段包括：

- `weakTopics`
  - 薄弱点列表。

- `nextRoundPriority`
  - 下一轮训练顺序。

- `practiceQuestions`
  - 可练习问题。

- `oneMinuteTemplates`
  - 1 分钟回答模板。

- `shouldRetry`
  - 是否建议重练。

### 2.5 前端展示

前端已经能展示：

- 历史面试复盘。
- 逐题学习复盘。
- 下一轮训练计划。
- 弱点和训练建议。

对应代码主要在：

- `app.js`
  - `createQuestionReviewCards()`
  - `createTrainingPlanSection()`
  - `renderReviewDetail()`
  - `renderReport()`

## 3. 为什么训练闭环对项目很重要

如果没有训练闭环，用户只能得到一次性反馈。

这种产品体验是：

```text
练一次
-> 看报告
-> 不知道下一步怎么练
```

有训练闭环后，体验变成：

```text
练一次
-> 知道哪道题没答好
-> 知道为什么没答好
-> 知道下一轮先练什么
-> 重新练弱点
-> 系统记住长期薄弱点
```

这才更像真实的面试训练产品。

## 4. 本项目训练闭环的数据流

可以用这条数据流理解：

```text
session.answers
-> /api/interview/report
-> build_question_reviews()
-> build_training_plan()
-> 前端报告区
-> 保存历史记录
-> 候选人画像 RAG
-> 下一轮 next-question
```

其中关键点是：

- `session.answers` 保存本轮所有问答。
- `build_question_reviews()` 保证每道题都有复盘，即使模型漏字段也有 fallback。
- `build_training_plan()` 从逐题复盘中提取弱点。
- 历史记录保存后，可以形成候选人画像。
- 下一轮 Agent 可以通过候选人画像 RAG 找到用户长期薄弱点。

## 5. 训练闭环和三个 RAG 的关系

三个 RAG 在训练闭环中的分工不同。

### 5.1 岗位知识库 RAG

回答：

```text
这个岗位应该考什么？
```

它决定训练内容是否贴合岗位。

### 5.2 题库 RAG

回答：

```text
这个考点有什么典型问法？
```

它决定问题是否像真实面试。

### 5.3 候选人画像 RAG

回答：

```text
这个人过去哪里薄弱？
```

它决定下一轮训练是否个性化。

面试时可以这样讲：

> 岗位知识库让问题贴合岗位，题库 RAG 让问法贴近真实面试，候选人画像 RAG 则让系统记住用户长期薄弱点，从而形成训练闭环。

## 6. 训练闭环和 Agent 的关系

Agent 不只是生成问题。

在训练闭环里，Agent 负责调度：

- 当前要不要继续追问？
- 用户连续答不上来要不要降难度？
- 本轮结束后是否建议重练？
- 下一轮训练优先级是什么？
- 候选人画像是否需要更新？

当前项目的 Agent trace 已经能看到：

```text
observe_state
-> analyze_answer
-> retrieve_context
-> select_action
-> generate_question
-> update_memory
```

其中 `update_memory` 当前是 `deferred`，表示系统已经记录了记忆更新意图，但真实长期画像更新更适合放在报告或历史保存阶段。

## 7. 当前项目还有哪些可补强点

当前训练闭环已经能用，但还可以继续工程化。

### 7.1 弱点标签标准化

现在弱点来自模型输出和 fallback。

后续可以把弱点标准化成标签：

- `rag_basic`
- `backend_architecture`
- `database_index`
- `agent_state`
- `deployment_nginx`

好处：

- 便于统计。
- 便于下一轮检索。
- 便于生成训练计划。

### 7.2 重练题生成

当前 `practiceQuestions` 是训练问题列表。

后续可以进一步做成：

- 一键开始弱点重练。
- 只练 RAG 薄弱点。
- 只练 FastAPI 后端结构。
- 只练项目讲解。

### 7.3 候选人画像更新

当前候选人画像主要来自历史记录和报告。

后续可以在保存历史记录时，结构化写入：

- 高频薄弱点。
- 高频风险点。
- 分数趋势。
- 最近训练建议。
- 已掌握考点。

### 7.4 下一轮问题优先级

下一轮 `next-question` 可以优先参考：

```text
trainingPlan.nextRoundPriority
候选人画像 RAG
最近一次 questionReviews
```

这样问题会更像持续训练，而不是每次从头开始。

## 8. 面试时怎么讲

可以这样讲：

> 我的系统不是一次性模拟面试，而是做了训练闭环。用户完成一轮面试后，后端会生成逐题复盘 questionReviews，每道题都会记录考察点、回答状态、缺失点、为什么问这题和下一步训练动作。然后系统会把逐题复盘汇总成 trainingPlan，包括薄弱点、下一轮训练顺序、练习题、1 分钟回答模板和是否建议重练。历史记录保存后，这些结果还能进入候选人画像，下一轮 Agent 再通过候选人画像 RAG 调整追问方向。

如果面试官问“这和普通报告有什么区别”，可以这样答：

> 普通报告只告诉用户分数和建议，我这里把报告拆成逐题复盘和下一轮训练计划。也就是说，系统不只是评价用户，而是指导用户下一轮怎么练。

如果面试官问“候选人画像 RAG 有什么用”，可以这样答：

> 候选人画像 RAG 让系统能记住用户长期薄弱点。比如用户多次在 RAG 质量评估、FastAPI 模块边界上回答不好，下一轮面试时 Agent 可以优先围绕这些点追问或给训练建议。

## 9. 下一步开发建议

Stage 3 后续可以按这个顺序开发：

1. 下一轮 `next-question` 更明确地利用候选人画像中的高频弱点标签。
2. 为不同 weakTags 配置固定训练题模板。
3. 后续把训练计划做成更完整的训练路径，例如“先看讲解 -> 再做复练 -> 再进真实面试模式”。

当前不建议一口气全做，因为训练闭环涉及报告、历史、候选人画像、RAG 和前端入口，改动范围比较大。更稳的方式是先标准化弱点标签，再逐步接入画像和重练入口。

## 10. 本阶段已完成：弱点标签标准化

本阶段新增了 `backend_python/training_tags.py`。

它负责把自然语言薄弱点归一成稳定标签。例如：

- `RAG 召回链路` -> `rag_retrieval`
- `Hit@K / MRR / 关键词覆盖率` -> `rag_quality`
- `Agent State / Orchestrator` -> `agent_state`
- `FastAPI router / schema / 接口` -> `backend_fastapi`
- `SQLAlchemy model / 外键 / relationship` -> `database_modeling`
- `Docker / Nginx / Uvicorn / Redis` -> `deployment_readiness`

现在报告接口返回的：

- `questionReviews[*].weakTags`
- `trainingPlan.weakTopics[*].weakTags`

都会带上标准化标签。

### 10.1 为什么弱点标签重要

如果只有自然语言弱点，系统很难统计：

```text
RAG 召回链路
检索增强生成
知识库命中
向量检索不理解
```

这些表达对人来说可能都是一类问题，但程序不知道它们属于同一个方向。

标准化标签可以让系统后续做到：

- 统计用户高频薄弱点。
- 把薄弱点写入候选人画像 RAG。
- 下一轮优先追问相同标签。
- 生成“一键重练某个弱点”的入口。
- 对不同用户做训练路径对比。

### 10.2 面试时怎么讲

可以这样讲：

> 我在训练闭环里加了弱点标签标准化。模型生成的复盘是自然语言，比如“RAG 召回链路不清楚”或“FastAPI 模块边界表达不完整”，但系统后续要统计和检索，就需要稳定标签。所以我用规则先把常见薄弱点归一成 `rag_retrieval`、`rag_quality`、`agent_state`、`backend_fastapi` 等标签，并写入 questionReviews 和 trainingPlan。这样后续候选人画像 RAG 和一键重练功能就有了结构化依据。

## 12. 本阶段已完成：一键重练薄弱点入口

本阶段继续把训练闭环从“能看到报告”推进到“能进入下一轮训练”。

前端报告里的 `trainingPlan` 现在会展示一个“一键重练薄弱点”按钮。用户点击后，系统会：

- 自动切换到 `coach` 学习辅导模式。
- 自动切换到 `quick` 快速训练强度。
- 根据薄弱点推断下一轮面试类型，例如 RAG、Agent、FastAPI 等技术薄弱点会进入“技术一面”。
- 把本轮 `trainingPlan` 里的薄弱点、薄弱原因、训练动作和推荐练习题写入下一轮 profile 的公司/岗位要求上下文。
- 复用现有 `/api/interview/next-question` 接口启动下一轮训练，不新增后端 API。

这一步的意义是：

```text
报告里的训练建议
-> 前端一键复练入口
-> 下一轮 coach 模式训练
-> 继续生成问答和报告
```

也就是说，训练报告不再只是静态结果，而是变成了下一轮训练的入口。

### 12.1 为什么不新增后端接口

当前阶段为了控制风险，没有新增 `/api/training/retry` 之类的接口，而是复用已有面试启动流程。

这样做有三个好处：

- 不破坏现有前后端协议。
- 不增加数据库迁移。
- 更容易解释：训练闭环先通过前端编排完成，后续再逐步抽象成后端训练任务。

### 12.2 面试时怎么讲

可以这样讲：

> 我把面试报告里的 trainingPlan 做成了可执行入口，而不是只展示建议。前端会根据 weakTopics 和 practiceQuestions 自动切换到学习辅导模式，把薄弱点写入下一轮 profile 上下文，然后复用 next-question 接口启动复练。这样用户从“看到薄弱点”到“马上开始针对性训练”之间不需要手动复制内容，产品闭环更完整。

如果面试官问“为什么这一步先放在前端做”，可以这样答：

> 因为当前目标是先完成 MVP 级训练闭环，不引入新的后端状态和数据库表。前端可以把已有 trainingPlan 转成下一轮面试输入，验证产品路径是否成立。等路径稳定后，再把复练任务沉淀成后端训练计划表或任务流。

## 11. 本阶段已完成：weakTags 进入候选人画像

本阶段继续把 `weakTags` 接入候选人画像。

现在历史报告中的：

- `questionReviews[*].weakTags`
- `trainingPlan.weakTopics[*].weakTags`

会在候选人画像检索时被提取出来，并进入：

- `memories[*].weakTags`
- `candidateProfile.frequentWeakTags`

同时，`format_candidate_memory()` 和 `format_candidate_profile()` 会把薄弱标签写入 prompt 友好的文本里。

### 11.1 为什么这一步重要

上一轮只是让报告里出现 `weakTags`。

这一轮让长期画像也知道这些标签，意义更大：

```text
一次报告里的弱点
-> 历史记录
-> 候选人画像
-> 候选人画像 RAG
-> 下一轮 Agent 决策和追问
```

这才真正把训练计划和下一轮面试连接起来。

### 11.2 面试时怎么讲

可以这样讲：

> 我进一步把 weakTags 接入了候选人画像。报告里每道题的 weakTags 会随着历史记录保存下来，后续候选人画像检索时会聚合出 frequentWeakTags。这样下一轮 Agent 不只是看到“用户过去分数低”，还能看到用户长期薄弱方向，比如 rag_quality、agent_state、backend_fastapi，从而更有针对性地追问或给训练建议。
