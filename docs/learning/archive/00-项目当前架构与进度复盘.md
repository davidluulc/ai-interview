# 00 项目当前架构与进度复盘

## 1. 当前项目定位

AI 模拟面试系统面向大学生、应届生和社会求职者，核心流程是：用户填写简历、岗位 JD 和公司要求，后端结合三类 RAG 和 Interview Orchestrator Agent 生成动态面试问题，面试结束后生成复盘报告和训练建议。

这个项目不是简单的聊天页面，而是一个 AI 应用工程化项目。它把用户系统、投递档案、RAG 检索、Agent 决策、日志可观测和训练报告串成了一条完整链路。

## 2. 当前核心模块

### 2.1 前端

前端主要由三个文件组成：

- `index.html`：页面结构，包括登录、投递档案、面试工作台、RAG 调试、Agent 日志和报告区域。
- `styles.css`：页面样式，包括工作台布局、聊天框、调试面板、日志卡片和移动端适配。
- `app.js`：前端交互逻辑，包括登录态、投递档案、面试流程、报告保存、RAG 调试和 Agent 调试展示。

当前前端没有使用 React、Vue 或 Next.js，而是原生 HTML/CSS/JavaScript。这对学习阶段有一个好处：你能更直接地理解请求如何发出、数据如何渲染、状态如何变化。

### 2.2 FastAPI 后端

后端入口是 `backend_python/main.py`，它负责创建 FastAPI 应用、注册路由、中间件和静态文件。

核心后端模块包括：

- `backend_python/routes/auth.py`：用户注册、登录、刷新 token、退出登录。
- `backend_python/routes/interview.py`：面试主流程，包括下一题生成、RAG 检索、Agent 决策和报告生成。
- `backend_python/routes/rag.py`：RAG 调试接口和 RAG 日志接口。
- `backend_python/routes/rag_documents.py`：知识库文档管理。
- `backend_python/routes/history.py`：历史面试记录。
- `backend_python/database.py`：数据库连接、会话和 SQLite 兼容建表。
- `backend_python/db_models.py`：数据库表模型。
- `backend_python/schemas.py`：接口请求和响应结构。

### 2.3 三类 RAG

当前系统有三类 RAG：

- 岗位知识库 RAG：存岗位、技术栈、业务场景、工程化知识。
- 题库 RAG：存可追问的问题、参考答案、评分点。
- 候选人画像 RAG：存候选人的简历、历史回答、薄弱点和训练建议。

这三类 RAG 的职责不同。岗位知识库解决“这个岗位应该考什么”，题库 RAG 解决“可以怎么问”，候选人画像 RAG 解决“这个人应该重点追问什么”。

### 2.4 RAG 工程化模块

RAG 相关工程化文件包括：

- `backend_python/retrieval_service.py`：BM25、向量检索、hybrid search、rerank 相关逻辑。
- `backend_python/rag_quality.py`：根据命中数量和分数生成 RAG 质量摘要。
- `backend_python/rag_evaluation.py`：Hit@K、MRR、关键词覆盖率、metadataMatch、emptyRecall 等评估指标。
- `backend_python/rag_evaluation_seed.py`：用于 RAG 评估的种子文档。
- `backend_python/rag_explain.py`：面向用户和开发者的 RAG 命中解释。
- `backend_python/rag_logging.py`：RAG 命中日志记录和序列化。

这说明当前项目已经不只是“能搜资料”，而是逐步具备了真实 RAG 项目里的可评估、可解释、可排查能力。

### 2.5 Agent 工程化模块

Agent 相关文件包括：

- `backend_python/interview_agent.py`：面试 Agent 的核心决策和问题生成。
- `backend_python/agent_state.py`：构造 Agent State。
- `backend_python/agent_tools.py`：封装工具调用。
- `backend_python/agent_trace.py`：记录节点链路。
- `backend_python/agent_logging.py`：记录 Agent 决策日志。
- `backend_python/agent_orchestrator.py`：编排下一题生成流程。

当前 Agent 已经支持 coach/interview 模式、连续弱回答识别、重复问题保护、topic shift、nodeTrace 和 debugSignals。

## 4. 为什么今晚优先补 RAG 工程化

今晚优先做 RAG 工程化，不是因为 Agent 不重要，而是因为 RAG 是当前项目最容易被面试深挖、也最容易拉开项目质量差距的一块。

面试官可能会问：

- 你的知识库数据从哪里来？
- 文档怎么切 chunk？
- metadata 有什么用？
- 怎么评估一次召回质量？
- Hit@K、MRR、关键词覆盖率分别是什么？
- RAG 命中不准时怎么排查？
- 为什么这个问题是基于这些资料问出来的？

如果只能回答“我用了 RAG 检索资料”，项目深度会显得不够。更好的回答是：系统有固定 seed 数据、evaluation case、质量指标、命中解释和前端调试面板，能证明 RAG 不是黑箱。

## 5. 当前风险记录

当前工作区存在大量未跟踪和已修改文件。这是项目迭代过程中的历史状态，不适合在本轮自动做 git commit。

本轮原则：

- 不回退用户已有修改。
- 不做无关目录清理。
- 不做大规模重构。
- 每个阶段通过测试后再进入下一阶段。

## 6. 面试时怎么讲

可以这样表达：

```text
我在上线部署前先做了一轮工程化体检。项目当前已经有 FastAPI 后端、用户系统、投递档案、三类 RAG、Agent 决策、历史记录和前端调试面板。

在继续开发前，我先跑了后端 pytest 和前端 .mjs 测试，确认基线稳定。然后我把下一阶段重点放在 RAG 工程化上，因为 RAG 的数据质量、召回质量和可解释性会直接影响 AI 面试官的问题质量。

这个阶段的目标不是盲目加功能，而是把 RAG 从“能检索”升级成“能评估、能解释、能展示质量”的工程化链路。
```
