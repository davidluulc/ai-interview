# 09 AI 模拟面试系统阶段性项目讲解

> 这份文档的目标不是继续堆功能，而是帮你把当前项目讲清楚。
> 你后面准备实习面试时，可以把它当成项目答辩稿、复盘提纲和追问清单。

## 1. 一句话介绍项目

这是一个面向大学生、应届生和社会求职者的 AI 模拟面试训练系统。

用户可以维护自己的求职档案，输入目标岗位 JD 和简历信息，系统会结合岗位知识库 RAG、题库 RAG、候选人画像 RAG 生成贴近岗位和个人经历的面试问题。每轮回答后，Interview Orchestrator Agent 会根据当前回答质量、历史薄弱点、RAG 命中结果和剩余轮次决定下一步是继续深挖、降低难度、切换话题、进入学习辅导，还是结束面试并生成报告。

面试表达可以这样说：

```text
我做的是一个 AI 模拟面试训练系统，核心不是简单调用大模型聊天，而是把求职训练拆成用户档案、三类 RAG、Agent 决策、面试问答、报告复盘和训练闭环几个模块。
系统会结合岗位 JD、简历和历史回答生成问题，并通过 Agent State、ToolCalls、Agent Decision、nodeTrace 和日志让整个 AI 决策过程可解释、可调试。
```

## 2. 项目解决谁的什么问题

目标用户主要有两类：

- 大三、大四准备找实习或校招的学生。
- 已经工作或转方向、需要练习项目表达和面试追问的求职者。

用户痛点是：

- 不知道真实面试会围绕简历和岗位怎么追问。
- 项目会写在简历上，但讲不清楚技术细节。
- 普通聊天式 AI 容易泛泛而谈，不能持续跟踪薄弱点。
- 不知道自己哪里弱，也不知道下一轮该怎么练。

本项目的产品价值是：

- 用 RAG 让问题更贴合岗位、题库和候选人历史。
- 用 Agent 决策让面试流程不只是固定题目列表。
- 用报告和 weakTags 把一次面试沉淀成下一轮训练目标。
- 用 RAG 日志和 Agent 日志减少黑箱感，方便开发者调试和面试时讲工程化能力。

## 3. 一次完整使用流程

从用户视角看，完整流程是：

```text
注册 / 登录
-> 创建或维护求职档案
-> 输入简历、目标岗位、JD、公司要求
-> 选择学习辅导模式或真实面试模式
-> 系统生成第一题
-> 用户回答
-> 后端调用三类 RAG
-> Agent 分析当前局面并做下一步决策
-> 大模型生成下一题
-> 多轮问答后生成报告
-> 报告中输出逐题复盘、风险点、训练建议、weakTags
-> 下一轮可以一键重练薄弱点
```

从后端视角看，核心链路是：

```text
前端提交回答
-> /api/interview/next-question
-> 读取当前用户和求职档案
-> 调用岗位知识库 RAG、题库 RAG、候选人画像 RAG
-> 计算 retrievalQuality
-> 构造 Agent State
-> Agent 选择下一步动作
-> normalize / fallback / guardrail 兜底
-> 生成下一道问题
-> 写入 Agent 决策日志和 RAG 命中日志
-> 返回前端展示
```

## 5. 后端整体架构怎么讲

你可以把 FastAPI 后端理解成分层结构：

```text
main.py
  创建 FastAPI 应用，注册中间件、异常处理和路由。

routes/
  接 HTTP 请求，做参数校验，调用业务模块，返回响应。

schemas.py
  定义请求体和响应体的数据结构。

db_models.py
  定义数据库表结构和表关系。

database.py
  定义数据库连接、Session、初始化和迁移兼容逻辑。

security.py
  处理密码哈希、token 生成、token 校验、当前用户获取。

agent_*.py / interview_agent.py
  处理 Agent 状态、工具调用、流程编排、决策和轨迹日志。

rag*.py / retrieval_service.py / rag_store.py
  处理 RAG 数据、检索、文档 chunk、embedding、hybrid 和 rerank。

candidate_memory.py / weakness_*.py
  处理候选人画像、历史薄弱点、训练策略和训练模板。
```

面试表达：

```text
我没有把所有逻辑写在一个接口函数里，而是按职责拆分。routes 负责接请求，schemas 负责数据校验，db_models 和 database 负责数据层，RAG 和 Agent 模块负责核心业务逻辑。这样做的好处是接口层更薄，业务模块更容易测试，也更方便后续迁移到 LangGraph 或生产部署。
```

## 6. 数据库表关系怎么讲

当前数据库的核心表大致可以按“用户为中心”理解。

### 6.1 User

`User` 是用户主表。

它关联：

- `RefreshToken`：用户登录续期。
- `ApplicationProfile`：用户的求职档案。
- `InterviewRecord`：用户的历史面试记录。
- `RagRetrievalLog`：用户触发的 RAG 命中日志。
- `AgentDecisionLog`：用户触发的 Agent 决策日志。
- `RagDocument`：用户上传或维护的知识库文档。
- `RagChunk`：用户知识库文档切分后的 chunk。

### 6.2 ApplicationProfile

`ApplicationProfile` 是求职档案。

它保存：

- 简历文本。
- 目标岗位。
- JD。
- 公司要求。
- 其他和当前投递有关的信息。

它和 `InterviewRecord` 的关系是：

```text
一个求职档案可以对应多次面试记录。
一条面试记录可以挂到某个求职档案上。
```

### 6.3 InterviewRecord

`InterviewRecord` 是历史面试记录。

它保存：

- 面试问题和回答。
- 报告 JSON。
- 分数。
- 风险点。
- 训练建议。
- 所属用户。
- 所属求职档案。

面试记录的价值不只是“历史列表”，还会进入候选人画像 RAG，成为后续训练的长期记忆来源。

### 6.4 RagDocument 和 RagChunk

`RagDocument` 是文档层，表示一份知识库文档。

它保存：

- 文档标题。
- 知识库类型，例如 `role_knowledge` 或 `question_bank`。
- 原始文本。
- 元数据。
- 所属用户。

`RagChunk` 是切片层，表示文档被切分后的片段。

它保存：

- chunk 内容。
- chunk 序号。
- metadata。
- embedding JSON。
- embedding 模型。
- embedding 状态。
- 所属文档。
- 所属用户。

这两个表的关系是：

```text
一份 RagDocument 可以切成多个 RagChunk。
RAG 检索时真正被召回的一般是 RagChunk。
```

### 6.5 外键和 relationship 怎么讲

外键解决的是：

```text
这条记录属于哪条父记录？
```

例如：

```text
interview_records.user_id -> users.id
```

意思是：这条面试记录属于 users 表里的某个用户。

但外键本身不会自动知道“当前登录用户是谁”。真正保证用户隔离的是业务代码：

```text
用户请求接口
-> get_current_user 解析 token
-> 后端拿到 current_user.id
-> 保存数据时写入 current_user.id
-> 查询数据时过滤 user_id == current_user.id
```

`relationship` 解决的是 Python 对象层导航：

```text
record.user
user.interview_records
```

面试表达：

```text
外键保证数据库关系合法，比如一条面试记录必须指向真实存在的用户；relationship 让 SQLAlchemy 在 Python 对象层可以顺着关系访问关联对象。真正的用户数据隔离由鉴权和查询条件保证，保存和查询时都使用当前 token 解析出的 current_user.id。
```

## 7. 三类 RAG 怎么协作

本项目不是把所有资料塞到一个知识库里，而是拆成三类 RAG。

### 7.1 岗位知识库 RAG

它回答的问题是：

```text
这个岗位需要什么能力？
```

它适合存：

- 岗位 JD。
- 技术栈要求。
- 业务场景。
- 岗位评分标准。
- 常见追问方向。
- 实习岗和初级岗常见考点。

它服务于：

- 让问题贴合岗位。
- 让报告评分有岗位依据。
- 让 Agent 知道应该围绕哪些能力深挖。

### 7.2 题库 RAG

它回答的问题是：

```text
围绕当前岗位和阶段，可以参考哪些真实面试题？
```

它适合存：

- 面试题。
- 参考答案。
- 答题要点。
- 难度。
- 阶段。
- 标签。

它服务于：

- 给问题生成提供题型参考。
- 避免大模型完全自由发挥。
- 报告复盘时检查答题要点覆盖情况。

### 7.3 候选人画像 RAG

它回答的问题是：

```text
这个用户过去暴露过哪些薄弱点？
```

它适合存：

- 历史面试回答。
- 历史报告风险点。
- 历史训练建议。
- weakTags。
- frequentWeakTags。
- 当前求职档案下的薄弱点。

它服务于：

- 个性化追问。
- 长期训练闭环。
- 避免每次面试都像第一次见到用户。

### 7.4 三类 RAG 为什么不合并

如果全放进一个知识库，短期能跑，但长期会难维护：

- 岗位资料、题库和用户历史的权限不同。
- metadata 过滤规则不同。
- 召回质量评估方式不同。
- 日志排查时很难知道是哪类资料出了问题。
- 候选人画像必须做用户隔离，不能和公共题库混在一起。

面试表达：

```text
我把 RAG 拆成岗位知识库、题库和候选人画像，是因为它们解决的问题不同。岗位知识库提供岗位标准，题库提供真实问题参考，候选人画像提供个性化历史记忆。拆开后更容易做权限隔离、metadata 过滤、日志排查和召回质量评估。
```

## 8. RAG 工程化当前做到什么程度

当前项目的 RAG 不是只有“把文本丢给模型”，已经有一些工程化要素：

- 文档和 chunk 两层结构。
- chunk metadata。
- 统一知识库类型常量。
- BM25 检索。
- embedding 向量检索。
- Hybrid Search。
- Rerank 重排。
- RAG 命中日志。
- RAG debug 接口。
- RAG 质量评估样例。
- Hit@K、MRR、关键词覆盖率等指标解释。

但也要讲清楚边界：

- 当前还不是完整企业级知识库平台。
- 没有接入 Qdrant、Milvus、pgvector 这类生产向量数据库。
- 没有完整异步入库任务队列。
- 没有管理员后台管理所有知识库。
- 没有完整监控告警体系。

面试表达：

```text
当前 RAG 的重点是先建立可解释、可评估、可降级的检索链路。我做了文档 / chunk 分层，BM25、向量检索、Hybrid Search 和 Rerank 的统一检索服务，并记录命中日志和评估指标。生产级向量数据库、异步入库和后台权限管理属于后续增强方向。
```

## 9. BM25、向量检索、Hybrid、Rerank 怎么讲

### 9.1 BM25

BM25 偏关键词检索，适合：

- 框架名。
- 字段名。
- 技术名词。
- 岗位标签。
- 明确的业务关键词。

优点：

- 可解释。
- 稳定。
- 适合小型知识库。
- 方便调试命中词。

### 9.2 向量检索

向量检索偏语义召回，适合：

- 用户表达不标准。
- 同义改写。
- 自然语言描述。
- 关键词没完全匹配但语义相关的内容。

风险是：

- 可能召回语义相近但业务不准确的内容。
- 对 embedding 模型和数据质量有依赖。
- 调试难度高于关键词检索。

### 9.3 Hybrid Search

Hybrid Search 是 BM25 + 向量检索的融合。

它的思路是：

```text
同时走 BM25 和 vector 两路召回
-> 对两路分数做归一化
-> 按 chunkId 去重合并
-> 计算 hybridScore
-> 返回综合排序结果
```

这样能兼顾：

- BM25 的关键词稳定性。
- 向量检索的语义召回能力。

### 9.4 Rerank

Rerank 是在初步召回之后做二次排序。

它不负责从全库里找资料，而是对候选 chunk 重新判断相关性：

```text
Hybrid Top K
-> rerank 模型重新打分
-> 按 rerankScore 排序
-> 返回更相关的结果
```

面试表达：

```text
我没有直接用向量检索替代 BM25，而是保留 BM25 作为可解释基线。Hybrid Search 会同时走 BM25 和向量检索，对分数做归一化后按 chunkId 去重融合。Rerank 放在召回之后，只对候选 chunk 做二次排序。如果向量或 rerank 服务失败，系统可以降级回 BM25，保证主流程可用。
```

## 10. RAG 质量评估怎么讲

RAG 质量评估不是让你主观感觉“这次好像更准了”，而是构造可验证样例。

一个评估 case 通常包含：

- query：测试问题。
- knowledgeBase：目标知识库。
- expectedTitles：期望命中的标题。
- expectedKeywords：期望覆盖的关键词。
- topK：检查前几个结果。

常见指标：

### 10.1 Hit@K

```text
前 K 条召回结果里有没有命中预期相关内容。
```

例如 Hit@3 = 1，表示前 3 条里至少有一条相关。

### 10.2 MRR

```text
第一个正确结果排得越靠前，MRR 越高。
```

如果第一条就是正确结果，分数最高；如果正确结果排到第三条，分数会降低。

### 10.3 关键词覆盖率

```text
召回内容覆盖了多少预期关键词。
```

它适合判断召回结果有没有覆盖关键技术点。

面试表达：

```text
我用小型评估集验证 RAG 质量。每条 case 包含 query、目标知识库、期望命中的标题和关键词，然后计算 Hit@K、MRR 和关键词覆盖率。这样能对比 BM25、Vector、Hybrid 和 Rerank 是否真的提升召回效果，而不是只靠主观体验判断。
```

## 11. Agent 模块整体怎么讲

本项目的 Agent 可以叫：

```text
Interview Orchestrator Agent
```

它不是单纯的聊天机器人，而是面试流程编排器。

普通 LLM 调用是：

```text
用户输入
-> 拼 prompt
-> 调模型
-> 返回结果
```

本项目 Agent 是：

```text
观察状态
-> 调用工具
-> 分析回答
-> 检索上下文
-> 选择动作
-> 生成问题
-> 更新记忆意图
-> 写日志
```

当前显式节点包括：

```text
observe_state
-> analyze_answer
-> retrieve_context
-> select_weakness_strategy
-> select_training_template
-> select_action
-> generate_question
-> update_memory
```

面试表达：

```text
我的 Agent 不是让大模型随便问，而是把面试拆成多个节点。它先观察当前状态，再调用 RAG 工具，然后分析用户回答，选择弱点训练策略和下一步动作，最后再生成自然语言问题。每个节点都会进入 nodeTrace，方便解释这一题为什么这么问。
```

## 12. Agent State、ToolCalls、Agent Decision 的关系

这三个概念一定要讲清楚。

### 12.1 Agent State

Agent State 是 Agent 做决策前看到的当前局面。

它包含：

- 用户档案。
- 历史问答。
- 上一轮回答。
- 当前轮次。
- 剩余轮次。
- agentMode。
- answerAnalysis。
- retrievalQuality。
- weaknessStrategy。
- trainingTemplateHint。
- nodeTrace。
- toolCalls。

### 12.2 ToolCalls

ToolCalls 是 Agent 调用工具的记录。

本项目里主要是三类 RAG 工具：

- `retrieve_role_knowledge`
- `retrieve_question_bank`
- `retrieve_candidate_memory`

它记录：

- 工具名。
- 输入摘要。
- 输出摘要。
- 是否成功。
- 错误信息。
- 耗时。

### 12.3 Agent Decision

Agent Decision 是 Agent 对下一步动作的选择。

它包含：

- `nextAction`：下一步动作。
- `difficulty`：难度。
- `stage`：阶段。
- `focus`：关注点。
- `reason`：原因。
- `tools`：参考了哪些工具。
- `weaknessStrategy`：是否围绕薄弱点训练。
- `trainingTemplateHint`：具体训练模板。
- `fallbackUsed`：是否兜底。
- `guardrailApplied`：是否触发保护规则。

三者关系：

```text
ToolCalls 提供外部资料
-> Agent State 汇总当前局面
-> Agent Decision 决定下一步怎么问
-> generate_question 根据 Decision、RAG context 和 history 生成最终问题
```

面试表达：

```text
ToolCalls 是工具调用记录，Agent State 是决策输入，Agent Decision 是下一步动作方案。最终问题不是 Agent Decision 本身，而是问题生成逻辑结合 Decision、RAG 上下文和历史问答生成出来的自然语言问题。
```

## 13. fallback、normalize、guardrail 怎么讲

这三个词容易混，面试时可以这样区分。

### 13.1 fallback

fallback 是兜底方案。

当模型不可用、输出不合法，或者规则需要优先生效时，系统用预设规则生成一个能工作的决策。

例如：

```text
剩余轮次为 0 -> 结束面试。
连续答不上来 -> 降低难度或切换话题。
```

### 13.2 normalize

normalize 是格式校验和修正。

大模型可能输出不合法字段，normalize 会把它修正成系统允许的值，避免后续代码崩溃。

### 13.3 guardrail

guardrail 是行为护栏。

它解决：

- 不要重复问同一题。
- 不要一直卡在同一个话题。
- coach 模式下不要过度施压。
- interview 模式下可以保持压力，但不能无意义死磕。

面试表达：

```text
fallback 保证系统有兜底决策，normalize 保证模型输出格式合法，guardrail 保证 Agent 行为不跑偏。三者配合，让大模型参与决策时仍然保持稳定、可控、可观测。
```

## 14. coach 模式和 interview 模式怎么讲

项目现在支持两种 Agent 模式。

### 14.1 coach 模式

coach 是学习辅导模式。

特点：

- 更偏解释。
- 降低难度更积极。
- 用户答不上来时会拆小问题。
- 更适合补基础和训练表达。

适合场景：

- 用户还没学懂某个概念。
- 用户想准备项目讲解。
- 用户连续答不上来。

### 14.2 interview 模式

interview 是真实面试模式。

特点：

- 更像面试官追问。
- 有一定压力。
- 关注是否能给出准确、可验证的回答。
- 但仍然有重复问题保护和 topic shift 规则。

适合场景：

- 用户想模拟真实面试。
- 用户已经复习过，想检验掌握程度。
- 用户要练被追问时的表达。

面试表达：

```text
我把 Agent 分成 coach 和 interview 两种模式。coach 负责学习辅导，用户答不上来时更倾向于降难度和解释；interview 负责真实面试压力，会继续追问细节，但也通过 guardrail 避免机械重复和无意义卡死。
```

## 15. 训练闭环怎么讲

训练闭环是当前项目比普通模拟面试更有价值的地方。

一次面试结束后，系统不只是生成一个分数，而是生成：

- 总体评分。
- 风险点。
- 改进建议。
- 逐题复盘。
- 每题 weakTags。
- 下一轮训练计划。

这些 weakTags 会进入候选人画像：

```text
questionReviews[*].weakTags
-> trainingPlan.weakTopics[*].weakTags
-> candidateProfile.frequentWeakTags
-> weaknessStrategy
-> trainingTemplateHint
-> 下一轮问题生成
```

这意味着：

```text
一次面试暴露问题
-> 报告记录薄弱点
-> 候选人画像聚合薄弱点
-> 下一轮 Agent 优先训练薄弱点
```

面试表达：

```text
我做了一个训练闭环。报告不是一次性结果，而会把每题薄弱点沉淀成 weakTags。候选人画像会统计 frequentWeakTags，下一轮 Agent 会根据这些标签选择 weaknessStrategy 和 trainingTemplateHint，让训练能围绕用户长期薄弱点持续改进。
```

## 16. weakTag 训练模板怎么讲

weakTag 回答的是：

```text
用户哪里弱？
```

例如：

- `rag_quality`
- `rag_retrieval`
- `agent_state`
- `backend_fastapi`
- `database_modeling`
- `project_storytelling`

weaknessStrategy 回答的是：

```text
本轮要不要围绕这个弱点训练，以及采用什么策略？
```

trainingTemplateHint 回答的是：

```text
这个弱点具体怎么练？
```

它包含：

- 推荐问题。
- 答题要点。
- 常见错误。
- 当前模式。
- 推荐难度。

训练模板不是第四个 RAG。

区别是：

```text
题库 RAG：真实面试里可能问什么。
训练模板：某个候选人的长期薄弱点应该怎么练。
```

面试表达：

```text
我没有只记录 weakTag，而是把 weakTag 映射到训练模板。比如用户在 rag_quality 上长期薄弱，系统会选择 RAG 质量评估模板，里面包含 coach 和 interview 两种问法、答题要点和常见错误。这样下一轮问题生成不完全依赖模型自由发挥，而是被稳定的训练目标约束。
```

## 17. 日志和可观测性怎么讲

AI 应用最怕黑箱。

本项目主要做了两类日志。

### 17.1 RAG 命中日志

记录：

- query 文本。
- retrieverName。
- knowledgeBase。
- hitCount。
- hitsJson。
- retrievalMode。
- matchedTokens。
- bm25Score。
- vectorScore。
- hybridScore。
- rerankScore。
- quality 摘要。

作用是：

- 看这次到底召回了什么。
- 判断是否空召回。
- 判断是否召回错知识库。
- 对比 BM25、Vector、Hybrid、Rerank 的效果。

### 17.2 Agent 决策日志

记录：

- state_json。
- decision_json。
- nextAction。
- difficulty。
- focus。
- reason。
- fallbackUsed。
- guardrailApplied。
- nodeTrace。
- toolCalls。

作用是：

- 看 Agent 为什么这么问。
- 看是否触发了降难度、切话题或重复保护。
- 看模型决策是否被 fallback / normalize 修正。
- 给后续 LangGraph 迁移提供执行轨迹参考。

面试表达：

```text
我给 RAG 和 Agent 都做了可观测日志。RAG 日志解释检索命中了什么、为什么命中、分数如何；Agent 日志解释每一轮为什么选择这个动作、是否触发 fallback 或 guardrail。这样调试 AI 应用时不只看最终回答，而能定位是检索问题、状态问题、决策问题还是生成问题。
```

## 22. 面试官高频追问和回答模板

### 22.1 你这个项目和普通 ChatGPT 问答有什么区别？

回答：

```text
普通 ChatGPT 问答主要是用户输入后拼 prompt 生成回复。我的项目有明确的业务流程：先读取用户档案和岗位 JD，再通过三类 RAG 获取依据，然后 Agent 根据当前状态决定下一步动作，最后才生成问题。整个过程有 Agent State、ToolCalls、Decision 和日志记录，所以不是单轮聊天，而是带状态、工具和可观测性的面试流程编排。
```

### 22.2 为什么要三个 RAG？

回答：

```text
因为三类知识解决的问题不同。岗位知识库提供岗位能力标准，题库 RAG 提供真实面试题和答题要点，候选人画像 RAG 提供用户历史薄弱点和个性化记忆。拆开后更容易做权限隔离、metadata 过滤、质量评估和日志排查。
```

### 22.3 Agent State 里放什么？

回答：

```text
Agent State 是 Agent 的结构化上下文，里面包括用户档案、历史问答、上一轮回答、当前轮次、剩余轮次、agentMode、answerAnalysis、三类 RAG 的 retrievalQuality、weaknessStrategy、trainingTemplateHint、toolCalls 和 nodeTrace。它的作用是让 Agent 基于当前局面做决策，而不是只看用户最后一句话。
```

### 22.4 你怎么避免 Agent 一直问重复问题？

回答：

```text
我做了 answerAnalysis 和 guardrail。系统会根据历史问题判断 repeatedQuestionCount 和 topicLock，如果发现连续重复或者用户一直答不上来，就触发降难度或切换话题。coach 模式下更偏解释和引导，interview 模式下保持压力但不能无意义死磕。
```

### 22.5 RAG 检索质量怎么评估？

回答：

```text
我构建了小型评估集，每条 case 包含 query、目标知识库、期望命中的标题和关键词。系统会计算 Hit@K、MRR 和关键词覆盖率，用来判断前 K 条是否命中、正确结果排序是否靠前，以及关键知识点覆盖是否足够。
```

### 22.6 你项目里的训练闭环是什么？

回答：

```text
一次面试结束后，报告会生成逐题复盘和 weakTags。候选人画像会聚合历史 weakTags 得到 frequentWeakTags，下一轮 Agent 会根据这些薄弱点选择 weaknessStrategy 和 trainingTemplateHint，从而让下一轮问题更有针对性。这就形成了从面试暴露问题到下一轮训练改进的闭环。
```

### 22.7 为什么现在没有直接上 LangGraph？

回答：

```text
我没有一开始就引入 LangGraph，是因为我想先把 Agent 的底层概念做清楚，包括 state、tools、decision、fallback、guardrail 和 nodeTrace。当前自研 Orchestrator 代码量可控，也方便测试和讲解。后续迁移 LangGraph 时，可以把 observe_state、retrieve_context、select_action、generate_question、update_memory 映射成 graph nodes，把 nextAction 映射成 conditional edges，再用 checkpoint 保存多轮状态。
```

### 22.8 如果上线，你还要补什么？

回答：

```text
上线前需要补 Docker、Nginx、生产数据库、Redis、环境变量管理、日志落盘、健康检查、备份和监控。当前项目主要完成本地工程化闭环，后续会用 Nginx 做反向代理，Uvicorn 跑 FastAPI，数据库切到 PostgreSQL 或 MySQL，Redis 做限流、缓存和 token 黑名单。
```
