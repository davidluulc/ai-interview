# 01 RAG 知识库样例与评估数据集

## 1. 为什么 RAG 需要 seed 数据

seed 数据是项目初始知识库。它的作用不是把所有知识点都一次性塞进去，而是在用户还没有上传足够资料时，让系统已经具备一批可检索、可追问、可评分的基础知识。

在 AI 模拟面试系统里，seed 数据尤其重要：

- 岗位知识库 seed 可以告诉系统“这个岗位应该考什么”。
- 题库 seed 可以告诉系统“这些知识点可以怎么问”。
- evaluation seed 可以让测试脚本稳定复现检索效果。

没有 seed 数据，系统只能依赖用户输入和大模型通用能力，问题容易泛泛而谈。

## 2. 为什么 RAG 需要 evaluation case

evaluation case 是固定评估样例。它通常包含：

- `query`：要测试的检索问题。
- `knowledgeBase`：应该检索哪个知识库。
- `expectedTitle`：期待命中的资料标题。
- `expectedKeywords`：期待覆盖的关键词。
- `expectedPositionTag`：期待匹配的岗位标签。
- `expectedStage`：期待匹配的面试阶段。

它的价值是把“感觉能搜到”变成“可以被测试验证”。

比如一个 case 问：

```text
如何用 Hit@K、MRR 和关键词覆盖率做 RAG 质量评估？
```

系统应该能命中：

```text
RAG 质量评估与可观测面板
```

并覆盖：

```text
Hit@K、MRR、关键词覆盖率、调试面板
```

如果命不中，就说明知识库、query、检索器或 metadata 设计有问题。

## 3. 本阶段新增内容

本阶段新增了一小批上线前综合补强相关样例，而不是大规模灌入知识库。

新增方向包括：

- RAG Query Rewrite 与多路召回。
- Chunk 切分与 Metadata 设计。
- RAG 质量评估与可观测面板。
- Agent 与三类 RAG 协作。
- FastAPI 错误处理与请求日志。
- 上线前工程化准备。

这些内容都服务于一个目标：

```text
让项目在上线前具备更强的工程化可解释能力。
```

## 4. 本阶段测试做了什么

新增测试文件：

```text
tests/test_pre_deployment_rag_v2.py
```

它验证三件事：

- `data/role_knowledge_seed.json` 里存在新增岗位知识样例。
- `data/question_bank_seed.json` 里存在对应题库问题。
- `data/rag_evaluation_cases.json` 里存在对应评估 case。

同时还验证每条数据不是空壳：

- 关键词数量足够。
- 内容长度足够。
- 追问问题、评分点、风险信号完整。
- 题库问题有参考答案和 key points。

这就是“数据质量测试”。

## 5. 面试时怎么讲

可以这样讲：

```text
我没有把 RAG 知识库做成随便堆文本的形式，而是把知识库样例和评估样例绑定起来。

每条重要 seed 数据都要能对应一个 evaluation case。case 里会写 query、预期命中的标题、预期关键词、岗位标签和面试阶段。

这样我可以用自动化测试验证知识库是否真的能被召回，而不是只靠人工点页面看结果。
```

## 6. 这部分对项目有什么价值

这一步让 RAG 从“资料存进去”升级为：

```text
资料可维护；
召回可验证；
质量可测试；
面试可讲解。
```

这是 AI 应用工程化里很重要的一点：RAG 的质量不是靠感觉，而是靠样例、指标和测试逐步保证。
