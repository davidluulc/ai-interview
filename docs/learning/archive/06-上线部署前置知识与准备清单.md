# 06 上线部署前置知识与准备清单

## 1. 当前阶段定位

本阶段不做真实云服务器上线，不启动 Docker / Nginx / 云服务器。

目标是把上线前必须理解和准备的内容整理清楚，让项目具备“可上线规划”，而不是只停留在本地能跑。

你可以把这一阶段理解成：

```text
不上线，但先把上线前要检查什么、为什么检查、怎么检查讲清楚。
```

## 2. 一个 Web 后端上线后大概长什么样

典型链路：

```text
用户浏览器
-> 域名 / HTTPS
-> Nginx
-> Uvicorn
-> FastAPI 应用
-> 数据库
-> Redis / 对象存储 / 第三方模型 API
```

每一层职责不同：

- 浏览器：用户访问页面。
- 域名 / HTTPS：让用户通过公网安全访问。
- Nginx：公网入口、反向代理、HTTPS 终止、静态文件服务。
- Uvicorn：运行 Python ASGI 应用。
- FastAPI：处理业务接口。
- 数据库：保存用户、面试记录、RAG 文档、日志。
- Redis：后续可用于 token 黑名单、缓存、限流、任务状态。
- 对象存储：后续可保存 PDF / 图片简历。
- 模型 API：调用 Qwen / DashScope 等大模型能力。

## 3. Uvicorn 是什么

Uvicorn 不是云服务器，也不是 Linux 系统。

Uvicorn 是一个 Python ASGI 应用服务器。

它负责把 HTTP 请求转交给 FastAPI 应用。

本地启动命令：

```text
python -m uvicorn backend_python.main:app --host 127.0.0.1 --port 8000 --reload
```

可以理解成：

```text
Uvicorn = 跑 FastAPI 的 Python 服务器进程
```

上线时通常不会用 `--reload`，因为它是开发模式。

## 4. Nginx 是什么

Nginx 是公网入口层。

它常做这些事：

- 监听 80 / 443 端口。
- 配置 HTTPS 证书。
- 把请求转发给本机 Uvicorn。
- 服务静态文件。
- 做请求大小限制。
- 做超时控制。
- 后续可做限流和访问日志。

反向代理可以这样理解：

```text
用户只认识 Nginx，
Nginx 再替用户把请求转交给后面的 FastAPI。
```

所以反向代理不是“代理用户访问外网”，而是“代理后端服务接收公网请求”。

## 5. 数据库上线前要准备什么

当前本地默认 SQLite。

上线更推荐 PostgreSQL 或 MySQL。

上线前要确认：

- `DATABASE_URL` 不写死在代码里。
- 数据库账号密码来自环境变量。
- 生产数据库能连接。
- Alembic migration 能执行。
- 表结构和代码模型一致。
- 有备份策略。
- 敏感数据不直接打印到日志。

Alembic 的作用：

```text
管理数据库表结构版本。
```

它类似数据库结构的 Git。

## 6. Redis 上线前什么时候需要

当前项目不强制 Redis。

后续可以用于：

- refresh token / access token 黑名单。
- 限流。
- 缓存 RAG 检索结果。
- 异步任务状态。
- 防止重复提交。

现在不急着引入 Redis，是因为当前核心目标是把 FastAPI、RAG、Agent、数据库主链路打稳。

## 7. Docker 上线前什么时候需要

Docker 的价值是把运行环境打包。

它解决：

- 本机能跑，服务器跑不了。
- 依赖版本不一致。
- 部署步骤太复杂。
- 多服务启动顺序混乱。

但 Docker 不是第一天必须上。

当前阶段先把：

- 环境变量。
- 数据库迁移。
- 日志。
- 启动命令。
- 健康检查。

这些基础准备好，再做 Docker Compose 更稳。

## 8. 日志上线前要检查什么

上线后不能靠肉眼看页面调试。

至少要有：

- 请求日志。
- 错误日志。
- LLM 调用日志。
- RAG 命中日志。
- Agent 决策日志。
- 接口耗时。
- 异常 trace id。

当前项目已经具备：

- 请求日志中间件。
- 统一错误响应。
- RAG 命中日志。
- Agent 决策日志。
- LLM 调用工程化基础。

后续上线前可以补：

- 日志文件轮转。
- 结构化 JSON 日志。
- 错误告警。
- 慢接口记录。

## 9. 环境变量上线前要检查什么

敏感配置不能写进代码。

至少包括：

- `DASHSCOPE_API_KEY`
- `SECRET_KEY`
- `DATABASE_URL`
- `ACCESS_TOKEN_EXPIRE_MINUTES`
- `REFRESH_TOKEN_EXPIRE_DAYS`
- `LLM_TIMEOUT_SECONDS`
- `LLM_MAX_RETRIES`
- CORS 允许域名
- 文件上传大小限制

`.env` 不应该提交 Git。

`.env.example` 可以提交，用来告诉别人需要哪些变量。

## 10. 健康检查是什么

健康检查用于判断服务是否活着。

当前项目已有：

```text
GET /api/health
```

上线后 Nginx、监控平台或运维脚本可以访问这个接口，判断后端是否正常。

更完整的健康检查可以包含：

- 应用是否启动。
- 数据库是否可连接。
- 关键配置是否存在。
- 模型 API 是否可用。

## 11. 上线前不要遗漏什么

上线前要确认：

- 本地全量测试通过。
- `.env.example` 完整。
- 生产 `SECRET_KEY` 已替换。
- 数据库迁移能执行。
- 默认管理员账号不会硬编码。
- API key 不出现在 Git 里。
- 日志不输出密码和 token。
- 文件上传有限制。
- CORS 不要无脑放开所有域名。
- Nginx 超时要适配 LLM 慢请求。
- 有最基本的备份和回滚方案。

## 12. 面试时怎么讲

可以这样讲：

> 当前项目还没有直接部署到云服务器，但我已经按上线前工程化做了准备。项目本地由 Uvicorn 运行 FastAPI，未来公网入口会由 Nginx 负责反向代理和 HTTPS，数据库从 SQLite 切换到 PostgreSQL 或 MySQL 时通过环境变量和 Alembic migration 管理表结构。日志层面已经有请求日志、RAG 命中日志、Agent 决策日志，后续上线前会补日志轮转、备份、健康检查和 Nginx 超时配置。

如果面试官问“为什么不直接 Docker 上线”，可以答：

> 我没有一开始就把复杂度堆到 Docker 和云服务器上，而是先把本地工程化能力补齐，包括测试、配置、数据库迁移、日志和健康检查。等这些稳定后，再用 Docker Compose 管理 FastAPI、数据库、Redis 和 Nginx 会更稳。
