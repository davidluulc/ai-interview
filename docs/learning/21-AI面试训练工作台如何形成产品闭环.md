# 21 AI 面试训练工作台如何形成产品闭环

## 1. 什么是产品闭环

产品闭环不是页面上功能多，而是用户完成一个核心目标时，每一步都有明确状态和下一步动作。

在 AI 模拟面试系统里，用户的核心目标不是“和大模型聊天”，而是：

```text
准备一次面试
-> 完成一次训练
-> 看懂自己哪里弱
-> 针对薄弱点继续练
-> 再进行下一次模拟面试
```

所以面试工作台不能只停留在聊天框。它需要让用户知道：

- 面试前：这场训练要练什么。
- 面试中：当前进行到第几题。
- 面试后：下一步应该怎么训练。

## 2. 为什么 AI 应用不能只讲 RAG 和 Agent

RAG 和 Agent 是底层能力，但用户感受到的是产品流程。

如果只有 RAG：

```text
系统能查到资料，但用户不知道为什么这样问。
```

如果只有 Agent：

```text
系统能做决策，但用户不知道当前训练进入了什么阶段。
```

如果只有报告：

```text
用户能看到结果，但不知道接下来该练什么。
```

所以本项目把 RAG、Agent、报告和训练串起来：

```text
RAG 提供资料依据
Agent 决定追问策略
前端展示进度和解释
报告总结薄弱点
训练中心承接后续练习
```

这就是 AI 应用工程化里的“能力产品化”。

## 3. 面试前、面试中、面试后的状态流转

### 面试前

用户先选择投递档案，再配置本次面试：

- 模式：学习辅导 / 真实面试。
- 轮数：5 / 8 / 10。
- 难度：基础 / 标准 / 压力。
- 重点方向：项目深挖 / 技术基础 / RAG & Agent / 行为面试 / 综合。

这些配置让用户知道自己不是随便开始聊天，而是在做一场有目标的训练。

### 面试中

前端展示：

- 第几题 / 共几题。
- 当前模式。
- 当前难度。
- 当前重点方向。
- 本题为什么这样问。
- 本题参考了哪些资料。

这样用户不会迷失在多轮问答里，也能理解 AI 面试官的追问逻辑。

### 面试后

用户点击“结束并复盘”后进入历史和报告链路。报告页展示：

- 总体表现。
- 逐题复盘。
- 建议优先训练的 weakTag。
- 生成专项训练任务。
- 再来一场。

这样一次面试不会停在“答完就结束”，而是自然进入下一轮训练。

## 4. 本项目如何把 RAG、Agent、报告和训练串起来

本项目当前的闭环可以理解为：

```text
投递档案
-> 面试配置
-> /api/interview/next-question
-> RAG 检索上下文
-> Agent 决策下一步追问
-> 前端展示问题、进度和解释
-> 历史记录与报告
-> weakTag 训练任务
-> 再来一场
```

这次 Interview Workbench Experience V4 主要做前端体验层增强：

- `interview store` 负责保存本次面试配置和进度状态。
- `InterviewSessionSetup` 负责面试前配置。
- `InterviewProgressStrip` 负责面试中进度反馈。
- `InterviewFinishPanel` 负责结束复盘入口。
- `ReportPage` 负责把报告和训练任务串起来。

它没有重写后端 API，也没有重写 RAG / Agent / LangGraph。这样做的好处是风险小、边界清楚、容易测试。

## 5. 面试时怎么讲

可以这样表达：

```text
我在项目里没有只做一个 AI 聊天框，而是把模拟面试设计成了训练闭环。

用户先选择投递档案，然后配置本次面试的模式、轮数、难度和重点方向。面试过程中，前端会展示当前轮次、训练模式、考察重点，并结合 Agent 决策摘要和 RAG 命中资料解释为什么这样问。面试结束后，用户可以进入报告页查看薄弱点，再根据 weakTag 生成专项训练任务，最后回到面试台再来一场。

这个设计体现的是 AI 应用产品化能力：底层 RAG 和 Agent 负责智能决策，上层前端负责把状态、解释、复盘和训练路径呈现给用户，形成可持续迭代的训练流程。
```

如果面试官继续追问“为什么不直接改后端”，可以这样回答：

```text
这个阶段的目标是体验闭环，不是算法升级。后端的 RAG、Agent 和报告能力已经存在，所以我优先通过 Vue3 store 和组件拆分，把已有能力组织成更清晰的用户流程。这样能控制改动范围，也能避免在体验层迭代时破坏已有后端接口。
```
