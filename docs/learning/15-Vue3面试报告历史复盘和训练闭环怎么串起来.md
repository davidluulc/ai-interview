# 15. Vue3 面试报告、历史复盘和训练闭环怎么串起来

更新时间：2026-06-12

## 1. 这一阶段解决什么问题

之前 Vue3 用户端已经有档案页、面试页和训练页，但它们更像三个独立入口。

这一阶段开始把它们串成闭环：

```text
历史复盘页
-> 点击某一场面试
-> 进入面试报告页
-> 查看分数、逐题复盘、薄弱点和提问依据
-> 点击薄弱点
-> 进入训练中心
-> 开始、完成、归档训练任务
```

这样项目就不再只是“AI 问几个问题”，而是一个能帮助求职者持续训练的产品。

## 2. 为什么要拆 API、Store、Page、Component

Vue3 前端里可以粗略分成四层：

```text
API client：只负责和后端通信
Pinia store：负责保存状态和动作
Page：负责组织一个完整页面
Component：负责复用的局部 UI
```

对应到本阶段：

- `frontend/src/api/history.ts`：封装 `/api/history`。
- `frontend/src/stores/history.ts`：保存历史记录列表、加载状态、错误信息。
- `frontend/src/pages/app/HistoryPage.vue`：展示历史复盘列表。
- `frontend/src/stores/report.ts`：根据 recordId 找到某一场报告。
- `frontend/src/pages/app/ReportPage.vue`：展示单场报告详情。
- `frontend/src/api/training.ts`：封装训练任务接口。
- `frontend/src/stores/training.ts`：保存训练任务，并提供开始、完成、归档动作。
- `frontend/src/components/training/TrainingTaskList.vue`：展示训练任务卡片和按钮。
- `frontend/src/pages/app/TrainingPage.vue`：组织训练中心页面。

这种拆法的好处是：

- 后端接口变了，优先改 API client。
- 页面需要多个状态，优先放到 store。
- 视觉结构变了，优先改 page/component。
- 测试可以一层一层写，不用一上来测完整页面。

## 3. 历史复盘页怎么工作

历史页的核心数据来自：

```text
GET /api/history
```

前端通过 `listHistory()` 请求接口，然后 `useHistoryStore()` 保存结果。

页面 `HistoryPage.vue` 做三件事：

1. 页面加载时调用 `history.loadHistory()`。
2. 把每条历史记录渲染成卡片。
3. 点击“查看报告”时跳转到：

```text
/vue/app/reports/:recordId
```

这里的 `recordId` 就是后端保存的面试历史记录 id。

## 4. 报告页怎么工作

当前后端还没有单独的：

```text
GET /api/history/{id}
```

所以报告页暂时复用历史列表接口：

```text
GET /api/history
-> 前端根据 recordId 从列表里找到目标记录
```

这不是最终形态，但它保持了后端接口兼容，不需要为了第一版页面新增接口。

报告页主要展示：

- 当前档案。
- 目标岗位。
- 综合分数。
- 表现等级。
- 总结。
- 优势与风险。
- 逐题复盘。
- weakTags。
- Agent/RAG 提问依据摘要。

点击某个 weakTag 时，会跳到训练中心：

```text
/vue/app/training?recordId=12&weakTag=agent_tool_calling
```

这样训练页就知道用户是从哪一场报告、哪一个薄弱点过来的。

## 5. 训练任务怎么工作

训练任务来自：

```text
GET /api/training/tasks
```

后端任务状态是：

```text
todo
in_progress
done
archived
```

前端展示成：

```text
todo -> 待训练
in_progress -> 训练中
done -> 已完成
archived -> 已归档
```

训练中心支持三个动作：

```text
POST /api/training/tasks/{task_id}/start
POST /api/training/tasks/{task_id}/complete
POST /api/training/tasks/{task_id}/archive
```

对应到用户行为就是：

- 开始训练。
- 标记完成。
- 归档任务。

## 6. 为什么这一阶段是产品闭环

如果只有面试功能，用户问完就结束了。

如果有报告但没有训练，用户知道问题在哪里，却不知道下一步练什么。

如果有训练但没有来源，用户不知道训练任务为什么出现。

所以完整闭环应该是：

```text
面试暴露问题
-> 报告解释问题
-> weakTag 标记问题
-> 训练任务修复问题
-> 下一轮面试验证提升
```

这就是 AI 模拟面试系统比普通聊天机器人更像产品的地方。

## 7. 面试时可以怎么讲

可以这样表达：

```text
我在 Vue3 前端里把面试历史、报告详情和训练任务拆成了独立模块。
API client 负责请求 FastAPI 后端，Pinia store 管理历史列表、报告详情和训练任务状态，页面组件负责展示和路由跳转。
用户完成面试后，可以在历史复盘页进入单场报告，报告里展示分数、逐题复盘、薄弱标签和提问依据。
这些薄弱标签会继续驱动训练任务，用户可以从报告页生成专项训练任务，也可以在训练中心开始、完成或归档任务。
所以系统形成了从模拟面试到复盘，再到专项训练的闭环。
```

如果面试官追问为什么不直接在页面里写 fetch，可以回答：

```text
我没有把请求直接写死在页面里，因为页面只应该关心展示和交互。
API client 统一封装后端请求，store 统一管理状态，这样后续接口路径、错误处理、token refresh 或状态结构变化时，不需要到每个页面里到处改。
```

## 8. 当前还可以继续增强什么

本阶段已经完成了基础闭环：

- 历史页支持按档案、岗位关键词和时间排序筛选。
- 报告页支持根据本报告生成训练任务。
- 报告页会携带 `recordId` 和 `weakTag` 跳转训练中心。
- 训练页会根据 `recordId` 和 `weakTag` 自动筛选任务。
- 训练任务可以回到来源报告。

后续还可以继续做：

- 新增单独的报告详情后端接口，避免每次报告页都拉全量历史列表。
- 把逐题复盘做得更结构化，例如每题都有评分、改进示例和推荐训练。
- 在训练完成后，把掌握度变化反向写入候选人画像。
