# AI 模拟面试系统 3 天接管冲刺路线

更新时间：2026-06-14

## 1. 这份文档解决什么问题

这份文档不是继续规划新功能，而是帮助你快速把当前项目从“能运行”变成“我能讲、能改、能排查”。

你现在不需要从第一个文件第一行开始读代码。那样效率很低，也容易被庞大的项目文件夹吓住。更合理的方式是先抓住主干：

```text
业务全貌
-> 一次面试请求主链路
-> RAG / Agent / LangGraph 三个核心模块
-> FastAPI / 数据库 / Vue3 的工程结构
-> 简历表达和面试追问
```

目标不是三天成为架构师，而是三天达到“可以先投简历，并且能讲清项目主线”的水平。

## 2. 三天冲刺的最终目标

三天结束后，你至少要能做到：

- 用 2 分钟讲清楚项目是什么、给谁用、解决什么痛点。
- 画出一次“提交回答 -> 生成下一题”的核心链路。
- 讲清三个 RAG 分别存什么、为什么不是一个 RAG。
- 讲清 Agent 和普通 LLM 调用的区别。
- 讲清为什么引入 LangGraph，以及为什么不直接全量替换 classic Agent。
- 讲清后台可观测性：RAG 命中日志、Agent 决策日志、runtimeAudit。
- 说出 2 到 3 个技术难点和取舍。
- 写出第一版简历项目描述。

## 3. 不要求你三天做到什么

三天内暂时不要求：

- 读懂每一行源码。
- 能独立重写整个项目。
- 精通 LangGraph 源码。
- 精通 RAG 评估算法。
- 精通 Docker / Nginx / Redis / Celery。
- 能回答所有八股文。

这些可以后续边投边补。三天冲刺只解决一个问题：你能不能把这个项目讲成一个真实、可信、工程化的 AI 应用项目。

## 4. 每天的学习节奏

如果每天能学 4 到 6 小时，建议这样分配：

```text
2 小时：听讲和建立概念地图
1 小时：带着问题看源码定位
1 小时：自己复述
1 小时：整理面试表达
1 小时：模拟追问
```

如果每天只有 2 到 3 小时，就把三天计划拉长到 5 到 7 天，不需要硬撑。

## 6. 第 2 天：RAG 和 Agent

### 6.1 学习目标

第 2 天解决两个问题：

```text
RAG 怎样给面试官提供资料？
Agent 怎样决定下一题怎么问？
```

### 6.2 三类 RAG 怎么讲

```text
岗位知识库 RAG：
存岗位 JD、岗位技术栈、业务场景和岗位相关知识，用来让问题贴合目标岗位。

题库 RAG：
存常见面试题、考察点和参考问题，用来保证提问有面试语境。

候选人画像 RAG：
存候选人简历概况、历史回答、薄弱点和训练记录，用来让追问贴合个人经历。
```

为什么不是一个 RAG：

```text
因为三类资料的来源、用途和更新频率不同。岗位知识库偏岗位和业务，题库偏面试考察，候选人画像偏个人长期记忆。拆开以后，检索结果更容易解释，也方便分别记录命中质量。
```

```text
backend_python/rag_store.py

```

### 6.4 Agent 怎么讲

普通 LLM 调用：

```text
输入 prompt，模型直接返回文本。
```

本项目 Agent：

```text
先观察当前面试状态，再检索 RAG 上下文，再分析用户上一轮回答，最后决定下一步是继续追问、降低难度、切换话题还是结束面试。
```

Agent 核心对象：

```text
Agent State：
记录当前面试状态，包括 profile、history、RAG hits、回答状态、轮次、候选人画像等。

Tool Calls：
记录 Agent 使用了哪些工具，本项目里主要是三类 RAG 检索。

Agent Decision：
记录下一步动作，比如 deep_follow_up、lower_difficulty、switch_topic。

Agent Policy：
规则兜底层，用来处理弱回答、重复问题、话题锁、学习辅导/真实面试差异。

Agent Log：
把状态、决策、原因和工具调用写入日志，方便后台排查。
```

## 7. 第 3 天：LangGraph、后端工程化和简历表达

### 7.1 学习目标

第 3 天解决三个问题：

```text
LangGraph 在项目里到底做什么？
FastAPI 后端结构怎么讲？
简历上怎么写才不心虚？
```

### 7.2 LangGraph 怎么讲

```text
LangGraph 是 Agent 工作流框架，适合表达多节点、多状态、可 checkpoint 的 Agent 流程。但框架本身不保证模型输出质量，所以项目没有直接把 classic Agent 全量替换成 LangGraph，而是采用灰度迁移。
```

本项目当前做法：

```text
classic Agent 是默认稳定链路。
LangGraph 是管理员可选择的 canary 灰度链路。
普通用户默认不能直接使用实验链路。
LangGraph 输出必须经过 quality gate。
如果质量门禁不通过，就 fallback classic。
runtimeAudit 会记录 requestedRuntime、allowedRuntime、visibleRuntime、fallbackUsed 和原因。
```

### 7.4 FastAPI 后端怎么讲

你要掌握这些概念：

```text
main.py：
创建 FastAPI app，注册路由和中间件。

routes/：
按业务拆分接口，例如 auth、interview、rag_documents、admin、training。

schemas.py：
定义接口请求和响应的数据结构，用 Pydantic 做校验。

db_models.py：
定义数据库表结构，用 SQLAlchemy ORM 映射 Python 类和数据库表。

database.py：
定义数据库连接、SessionLocal 和 get_db。

Depends(get_db)：
给接口注入数据库会话。

Depends(get_current_user)：
给接口注入当前登录用户，同时完成鉴权。
```

### 7.5 数据库表关系先这样理解

```text
users：
用户主表。

application_profiles：
投递档案，属于某个用户。

interview_records：
面试历史，属于某个用户，也可以关联某个投递档案。

rag_documents：
知识库文档，属于某个用户。

rag_chunks：
文档切片，属于某个文档。

rag_retrieval_logs：
RAG 命中日志，用来记录某次请求查到了什么。

agent_decision_logs：
Agent 决策日志，用来记录 Agent 为什么这么问。

training_tasks：
训练任务，通常来自面试报告里的薄弱点。

langgraph_checkpoint_summaries：
LangGraph 运行状态摘要，用来做 runtime 审计和后台调试。
```

### 7.6 简历表达初稿

可以先写成：

```text
AI 模拟面试系统 | FastAPI + Vue3 + RAG + Agent + LangGraph

- 面向大学生和求职者设计 AI 模拟面试系统，支持投递档案、岗位 JD、简历信息、模拟面试、复盘报告和训练任务闭环。
- 设计三类 RAG：岗位知识库、题库、候选人画像，结合 BM25、向量检索、hybrid search、rerank 和命中日志，提高面试问题与岗位和个人经历的相关性。
- 实现 Interview Orchestrator Agent，通过 Agent State、Tool Calls、Agent Decision、Policy 和 Guardrail 控制追问、降难度、切换话题和重复问题保护。
- 引入 LangGraph canary 灰度链路，保留 classic Agent 默认稳定链路，通过 quality gate、fallback classic 和 runtimeAudit 保证实验链路可观测、可回退。
- 基于 Vue3 + Pinia + Vue Router 实现用户工作台和管理员后台，后台支持 RAG 质量诊断、Agent 决策日志、Runtime 对比和 Runtime 审计。

```

这版还可以继续压缩，后续根据你投递岗位再改。

## 10. 你如何诚实表达自己的贡献

你不需要假装每一行代码都是手写的。更好的说法是：

```text
这个项目是我在 AI 辅助开发模式下完成的。我主要负责业务方向选择、需求拆解、技术路线判断、功能验收、问题反馈和项目复盘。开发过程中我围绕 RAG、Agent、LangGraph 灰度、后台可观测性和训练闭环不断迭代需求，并通过测试和浏览器验证确认功能可用。
```

这不是减分项。AI 应用开发岗本来就需要会使用 AI 工具，但你必须证明自己不是只会复制粘贴，而是能判断、能验收、能解释、能继续维护。
