# 本地测试数据清理和演示账号固定化

## 1. 为什么要做这一步

项目开发到管理员后台阶段后，本地数据库里会积累大量测试账号、测试 RAG 文档、测试 RAG 日志和 Agent 决策日志。

这些数据本身不是 bug，因为它们通常来自自动化测试、接口调试和功能验证。但如果不治理，管理员后台就会被测试数据刷屏，看起来不像一个正式产品，也会影响后续演示和学习。

所以这一轮做了本地数据治理：

- 清理明显的测试账号和关联数据。
- 固定两个本地演示账号。
- 清理前备份 SQLite 数据库。
- 清理脚本默认 dry-run，避免误删。

## 2. 为什么不能手动乱删数据库

数据库里的数据通常存在关联关系。

比如一个用户可能关联：

- refresh token。
- 投递档案。
- 面试记录。
- RAG 文档。
- RAG chunk。
- RAG 检索日志。
- Agent 决策日志。
- 训练任务。

如果只手动删除 `users` 表里的某一行，而不处理关联数据，就可能留下孤儿数据，导致后台统计、列表展示或后续查询变得混乱。

所以更稳妥的方式是写脚本，由脚本按固定顺序清理关联表。

## 3. dry-run 是什么

dry-run 的意思是“预演执行”。

这次新增的脚本：

```powershell
python scripts/cleanup_dev_test_data.py
```

默认只统计将要清理的数据，不会删除。

输出示例：

```text
Mode: DRY-RUN
Matched test users: 6745
RAG documents: 2929
RAG retrieval logs: 4484
Agent decision logs: 1367
No data was deleted.
```

只有显式加上：

```powershell
python scripts/cleanup_dev_test_data.py --apply
```

才会真正删除匹配到的测试数据。

如果要把某个历史演示账号也纳入清理，可以显式传入：

```powershell
python scripts/cleanup_dev_test_data.py --include-email admin@ai-interview.com
```

确认 dry-run 输出只匹配预期账号后，再执行：

```powershell
python scripts/cleanup_dev_test_data.py --include-email admin@ai-interview.com --apply
```

这种设计的好处是：默认策略保守，不会误删真实账号；需要清理历史账号时，又可以通过显式参数完成。

脚本里只保存邮箱、用户名和角色，不硬编码密码。

设置密码时通过命令行参数临时传入：

```powershell
python scripts/seed_demo_accounts.py --password 你的本地演示密码
```

本地开发阶段可以使用统一演示密码；真正上线前必须换成强密码，并且数据库里只能保存密码哈希，不能保存明文密码。

## 5. 这轮做了哪些脚本

### 5.1 cleanup_dev_test_data.py

路径：

```text
scripts/cleanup_dev_test_data.py
```

作用：

- 识别明显测试账号。
- 默认 dry-run。
- `--apply` 时删除测试账号及关联数据。
- 默认保护指定演示邮箱。

识别测试账号的依据包括：

- `@example.com`。
- `codex_`、`summary_admin_`、`rag_quality_` 等测试用户名模式。
- `summary-admin-`、`admin-list-`、`rag-quality-` 等测试邮箱前缀。

### 5.2 seed_demo_accounts.py

路径：

```text
scripts/seed_demo_accounts.py
```

作用：

- 创建或更新固定演示账号。
- 不存在就创建。
- 已存在就更新用户名、角色和密码哈希。
- 可重复运行，不会重复制造账号。

这种“重复运行结果仍然稳定”的脚本，叫幂等脚本。

## 6. 面试表达

可以这样讲：

> 项目开发过程中，自动化测试和接口调试会产生大量测试账号、RAG 文档、RAG 日志和 Agent 日志。为了让管理员后台更接近真实产品，我做了本地数据治理脚本。清理脚本默认 dry-run，只预览会删除哪些测试数据；真正删除必须显式加 `--apply`。清理前会先备份 SQLite 数据库，同时固定普通用户和管理员两个演示账号。这样既能保持本地环境干净，又避免误删真实演示账号。

## 7. 这体现什么工程能力

这一步体现的是开发环境治理能力。

它不是核心业务功能，但它能说明你知道：

- 测试数据和真实演示数据要区分。
- 删除数据库数据前要备份。
- 破坏性操作要有 dry-run。
- 脚本要可重复执行。
- 密码不能硬编码进代码。
- 管理员后台展示效果依赖干净的数据环境。
