# AI 模拟面试系统 RAG 工程化 OpenSpec

## 1. 文档目的

本文档用于约束 AI 模拟面试系统后续 RAG 模块的工程化开发范围、数据边界、召回链路、元数据设计、可观测性和质量评估方式。

本文档不是当前 MVP 的功能说明，而是后续迭代的工程化规格说明。开发时应优先保证链路可解释、数据可追踪、权限可控、评估可落地，再逐步引入向量数据库、reranker、异步入库等重型能力。

## 2. 背景与问题

当前系统已经具备三个轻量 RAG：

- 岗位知识库 RAG：基于岗位、JD、技术栈、面试阶段召回岗位知识、评分点和风险信号。
- 题库 RAG：基于岗位标签、面试阶段、关键词召回题目、参考答案和答题要点。
- 候选人画像 RAG：基于用户历史面试记录、回答内容、报告风险点和训练建议召回候选人薄弱点。

当前版本可以支撑 MVP 链路，但仍存在以下问题：

- 召回内容数量少，缺少文档管理能力。
- 知识条目元数据不完整，无法稳定做权限过滤、版本过滤、阶段过滤。
- 召回结果主要用于调试展示，尚未充分转化为用户可感知的提问依据和评分依据。
- 缺少 RAG 命中日志，无法追踪每个问题实际使用了哪些上下文。
- 缺少召回质量评估，无法判断问题体验一般是检索问题、prompt 问题，还是知识库质量问题。
- 候选人画像已经具备用户隔离，但还需要进一步按投递档案维度优先召回。

## 3. 总体目标

RAG 模块的目标不是简单证明“项目使用了 RAG”，而是让 AI 面试官在以下方面具备稳定提升：

- 问题更贴合目标岗位和 JD。
- 追问更贴合候选人简历、项目经历和历史回答弱点。
- 评分更贴合岗位能力点和题库答题要点。
- 报告能够解释问题依据、评分依据和训练建议来源。
- 开发者能够追踪一次问题生成过程中命中了哪些知识、题库和候选人历史。
- 系统能够对召回质量做基础评估，为后续向量检索和 reranker 提供验证依据。

## 4. 非目标

以下能力不作为下一阶段的强制目标：

- 不立即建设完整管理后台。
- 不立即引入 Milvus、Qdrant、pgvector 等持久化向量数据库。
- 不立即实现复杂语义切分和 OCR 流水线。
- 不立即接入 Ragas 等完整自动化评估框架。
- 不立即实现多租户企业级知识库权限模型。
- 不立即实现 Celery 异步入库和任务重试。

上述能力可以作为后续演进方向，但当前阶段应先完成可解释、可追踪、可评估的工程闭环。

## 5. RAG 子模块职责边界

### 5.1 岗位知识库 RAG

职责：

- 存储岗位相关技术栈、业务场景、能力要求、评分点和风险信号。
- 为问题生成提供岗位能力上下文。
- 为报告生成提供评分依据和风险判断依据。

典型数据：

- 后端开发岗位能力要求。
- AI 应用开发岗位能力要求。
- RAG 项目常见追问方向。
- FastAPI、数据库、缓存、鉴权、部署等技术知识点。
- 岗位 JD 中抽取出的关键技能和业务要求。

不负责：

- 不存储候选人个人简历。
- 不存储用户历史回答。
- 不直接决定最终面试问题，只提供上下文和依据。

### 5.2 题库 RAG

职责：

- 存储结构化面试题、参考答案、答题要点、难度、面试阶段和岗位标签。
- 为问题生成提供可参考题目。
- 为评分报告提供答题要点覆盖情况。

典型数据：

- 技术基础题。
- 项目深挖题。
- 行为面试题。
- 场景排查题。
- HR 和薪资规划题。

不负责：

- 不存储用户个人历史。
- 不替代模型生成问题。
- 不保证原题照搬，模型可以基于题库题目生成更贴合上下文的变体问题。

### 5.3 候选人画像 RAG

职责：

- 存储和召回当前用户历史面试记录、历史回答、报告分数、风险点和训练建议。
- 优先按当前投递档案召回候选人历史弱点。
- 当前投递档案历史不足时，回退到用户全局历史。
- 为追问生成提供个性化依据。

典型数据：

- 用户简历概况。
- 用户每轮回答内容。
- 历史报告评分。
- 高频风险点。
- 高频训练建议。
- 弱项面试阶段。

不负责：

- 不召回其他用户数据。
- 不召回无权限的投递档案数据。
- 不把候选人历史直接展示给其他用户或管理员普通视图。

## 6. 核心数据模型

### 6.1 知识文档

后续应统一抽象为 `knowledge_documents`。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 文档 ID |
| `source_type` | 来源类型，例如 role_knowledge、question_bank、candidate_memory、company_experience |
| `title` | 文档标题 |
| `content` | 原始内容或清洗后的正文 |
| `owner_user_id` | 所属用户，公共知识可为空 |
| `application_profile_id` | 所属投递档案，可为空 |
| `position_tag` | 岗位标签 |
| `interview_stage` | 面试阶段 |
| `difficulty` | 难度 |
| `visibility` | public、private、system |
| `source_uri` | 来源地址或文件名 |
| `version` | 文档版本 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 6.2 知识切片

后续应统一抽象为 `knowledge_chunks`。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 切片 ID |
| `document_id` | 所属文档 |
| `chunk_index` | 切片序号 |
| `content` | 切片内容 |
| `token_count` | 估算 token 数 |
| `keywords` | 关键词 |
| `metadata_json` | 扩展元数据 |
| `embedding_status` | pending、ready、failed |
| `created_at` | 创建时间 |

### 6.3 RAG 命中日志

下一阶段优先实现 `rag_retrieval_logs`，用于记录每次问题生成或报告生成实际使用了哪些召回结果。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 日志 ID |
| `user_id` | 当前用户 |
| `application_profile_id` | 当前投递档案 |
| `interview_record_id` | 所属面试记录，可为空 |
| `request_type` | next_question、report、debug |
| `query_text` | 检索 query |
| `retriever_name` | role_knowledge、question_bank、candidate_memory |
| `retrieval_mode` | keyword、bm25、vector、hybrid |
| `hit_count` | 命中数量 |
| `hits_json` | 命中条目摘要、分数和元数据 |
| `used_in_prompt` | 是否进入 prompt |
| `created_at` | 创建时间 |

该表是后续做召回质量评估的基础。

## 7. 元数据规范

每条可召回内容至少应具备以下元数据：

- `source_type`：区分岗位知识、题库、候选人画像、公司面经等来源。
- `position_tag`：用于岗位过滤。
- `interview_stage`：用于面试阶段过滤。
- `difficulty`：用于控制问题深度。
- `owner_user_id`：用于权限隔离。
- `application_profile_id`：用于投递档案维度召回。
- `visibility`：用于区分系统公共知识和用户私有知识。
- `version`：用于知识更新和回溯。
- `created_at` / `updated_at`：用于排序、回溯和增量更新。

元数据命名必须稳定，不得在不同模块中混用同义字段。例如：

- 统一使用 `position_tag`，不同时使用 `role_tag`、`job_tag`。
- 统一使用 `interview_stage`，不同时使用 `stage`、`category` 表示同一概念。
- 统一使用 `application_profile_id`，不使用 `profile_id` 表示投递档案。

## 8. 召回链路设计

### 8.1 Query 构造

问题生成时，query 应由以下信息组成：

- 当前用户 ID。
- 当前投递档案 ID。
- 目标岗位。
- 岗位标签。
- 当前面试阶段。
- 简历摘要。
- JD 摘要。
- 公司要求。
- 当前轮历史回答。
- 历史薄弱点摘要。

报告生成时，query 应额外包含：

- 本轮全部问题。
- 本轮全部回答。
- 本轮命中的岗位知识和题库要点。

### 8.2 召回顺序

下一阶段采用以下召回顺序：

1. 岗位知识库 RAG：按岗位标签、JD、面试阶段召回。
2. 题库 RAG：按岗位标签、面试阶段、难度召回。
3. 候选人画像 RAG：优先按当前用户 + 当前投递档案召回。
4. 候选人画像 RAG 回退：如果当前投递档案历史不足，则按当前用户全局历史召回。
5. 合并上下文：去重、截断、按来源分组。
6. 写入 RAG 命中日志。
7. 注入 prompt。

### 8.3 回退策略

候选人画像 RAG 的回退规则：

- 当前投递档案历史记录数量大于等于 2：优先使用当前档案历史。
- 当前投递档案历史记录数量小于 2：使用当前档案历史 + 用户全局历史。
- 用户全局历史为空：返回“暂无候选人历史画像”，不强行生成虚假历史。

岗位知识库 RAG 的回退规则：

- 命中岗位标签：优先返回岗位标签一致内容。
- 未命中岗位标签：使用 JD 和技术关键词召回。
- 仍无命中：返回默认通用面试能力点。

题库 RAG 的回退规则：

- 命中当前阶段题目：优先返回当前阶段题目。
- 当前阶段无命中：返回同岗位其他阶段题目。
- 同岗位无命中：返回通用题库题目。

## 9. Prompt 使用规范

RAG 结果进入 prompt 时必须按来源分组，避免模型混淆。

推荐格式：

```text
【岗位知识库 RAG 命中资料】
- 标题：
- 内容：
- 评分点：
- 风险信号：

【题库 RAG 命中资料】
- 题目：
- 答题要点：
- 难度：

【候选人画像 RAG 命中资料】
- 历史问题：
- 历史回答摘要：
- 历史风险点：
- 历史训练建议：
```

模型生成问题时必须满足：

- 问题应明确对应一个面试阶段。
- 问题应尽量引用简历、JD、历史弱点或题库要点之一。
- 不得编造候选人没有提供过的经历。
- 如果历史画像为空，不得假装候选人过去暴露过某个弱点。
- 技术问答和报告评分应保持低温度，追问表达可以保持自然。

## 10. 前端展示规范

RAG Debug 面板属于开发者调试功能，不应作为普通用户主要体验。

普通用户更需要看到：

- 考察点：这道题考察什么能力。
- 追问依据：为什么问这个问题。
- 答题要点：回答应该覆盖哪些关键点。
- 回答缺失点：当前回答少了什么。
- 下一步训练建议：如何改进。

建议前端分两层：

- 普通视图：展示考察点、追问依据、训练建议。
- 调试视图：折叠展示命中词、分数、retriever、metadata、prompt 片段。

## 11. RAG 质量评估

下一阶段先实现轻量评估，不立即引入完整评估框架。

### 11.1 召回侧指标

| 指标 | 含义 |
| --- | --- |
| `profile_hit_rate` | 当前投递档案历史是否被命中 |
| `position_tag_match_rate` | 命中内容是否匹配当前岗位标签 |
| `stage_match_rate` | 命中题目是否匹配当前面试阶段 |
| `empty_recall_rate` | 无召回结果占比 |
| `top_k_source_distribution` | Top K 结果来源分布 |

### 11.2 生成侧指标

| 指标 | 含义 |
| --- | --- |
| `context_usage_rate` | 生成内容是否使用了召回上下文 |
| `question_groundedness` | 问题是否能追溯到简历、JD 或 RAG 内容 |
| `scoring_coverage` | 报告是否覆盖召回评分点 |
| `hallucination_flag` | 是否出现无来源的候选人经历 |

### 11.3 人工评估维度

面试体验可以按以下维度人工打分：

- 岗位相关性。
- 简历相关性。
- 追问深度。
- 问题自然度。
- 技术准确性。
- 评分可解释性。
- 训练建议可执行性。

## 12. 知识库管理路线

### 12.1 第一阶段：结构化种子数据

目标：

- 保留当前 JSON 种子数据。
- 补齐元数据字段。
- 引入 RAG 命中日志。
- 支持开发者手动维护题库和岗位知识。

### 12.2 第二阶段：数据库化知识库

目标：

- 将岗位知识库和题库从 JSON 迁移到数据库表。
- 支持新增、编辑、删除、启用、停用。
- 支持按岗位标签、面试阶段、难度过滤。
- 支持命中次数统计。

### 12.3 第三阶段：文档解析与切片

目标：

- 支持 PDF、Markdown、Word、文本上传。
- 支持基础清洗。
- 支持固定长度切分和 Markdown 标题切分。
- 记录文档来源、版本、章节、页码。

### 12.4 第四阶段：向量检索与混合召回

目标：

- 引入 embedding。
- 接入 Chroma、pgvector 或 Qdrant。
- 支持 keyword + vector hybrid search。
- 支持 metadata filter。
- 支持 reranker。

### 12.5 第五阶段：评估与监控

目标：

- 基于 RAG 命中日志统计召回质量。
- 建立固定测试集。
- 对比关键词检索、向量检索、混合检索效果。
- 记录不同检索策略对问题质量的影响。

## 13. 权限与安全

RAG 模块必须遵守以下权限规则：

- 用户只能召回自己的候选人画像和投递档案历史。
- 公共岗位知识和公共题库可以被所有用户召回。
- 用户私有知识必须带 `owner_user_id`。
- 投递档案历史必须带 `application_profile_id`。
- Debug 接口不能泄露其他用户的命中内容。
- 日志中不得记录明文密码、API Key 或完整敏感简历原文。

## 14. 下一阶段开发任务

推荐下一阶段按以下顺序开发：

1. 新增 `rag_retrieval_logs` 表。
2. 在问题生成和报告生成接口中记录 RAG 命中日志。
3. 候选人画像 RAG 支持当前投递档案优先召回。
4. RAG Debug 面板区分普通解释信息和开发者调试信息。
5. 报告中增加“本轮追问依据”和“命中能力点”。
6. 为岗位知识库和题库补齐元数据字段。
7. 增加轻量召回质量统计接口。

## 15. 面试表达建议

可以这样描述本项目的 RAG 工程化思路：

> 我没有把 RAG 简单做成“上传文档然后问答”，而是结合 AI 模拟面试业务拆成岗位知识库 RAG、题库 RAG 和候选人画像 RAG。岗位知识库负责岗位能力点和评分标准，题库 RAG 负责阶段化题目和答题要点，候选人画像 RAG 负责用户历史回答和薄弱点召回。为了让 RAG 可维护、可评估，我设计了元数据规范、投递档案维度召回、RAG 命中日志和召回质量指标。后续可以基于这些日志对比关键词检索、向量检索和混合检索的效果，而不是盲目堆向量数据库。

## 16. 验收标准

RAG 工程化下一阶段完成后，应满足以下标准：

- 每次生成问题至少能记录一条 RAG 命中日志。
- RAG 命中日志能区分岗位知识库、题库和候选人画像来源。
- 候选人画像召回优先使用当前投递档案历史。
- 历史不足时能回退到用户全局历史。
- 普通用户能看到问题考察点和追问依据。
- 开发者能在调试视图看到命中词、命中分数和元数据。
- 后端测试覆盖用户隔离、投递档案隔离和空召回回退。
- 不出现跨用户数据泄露。
