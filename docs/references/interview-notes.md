# 项目面试讲解材料

这份文档用于帮助你以后在实习或工作面试中讲清楚这个项目。

## 一分钟项目介绍

这个项目是一个面向大学生和求职者的 AI 模拟面试系统。

用户可以创建一次简历投递档案，填写或上传简历，粘贴目标岗位 JD，然后开始模拟面试。AI 面试官会根据简历、岗位要求和用户前面的回答动态追问。面试结束后，系统会生成结构化反馈报告，并保存历史面试记录，方便用户复盘。

## 技术栈

- 前端：HTML、CSS、JavaScript。
- 后端：Python FastAPI。
- 大模型：阿里百炼 / DashScope / Qwen。
- 简历解析：PDF 使用 `pypdf`，图片简历使用视觉模型。
- 历史记录：FastAPI + SQLite，前端失败时可退回 `localStorage`。

## 为什么需要后端

前端不能直接调用大模型 API。

原因：

- API key 会暴露在浏览器里，不安全。
- prompt 逻辑应该统一放在后端管理。
- 后端可以校验请求和返回格式。
- 后端可以处理模型异常。
- 后续加日志、数据库、RAG 都更方便。

可以这样回答面试官：

> 我没有让前端直接调用大模型，而是用 FastAPI 做了一层后端代理。这样可以保护 API key，也能统一管理 prompt、错误处理和后续的数据存储。

## 后端模块结构

```text
config.py             读取 API key、模型名和项目路径
schemas.py            定义请求和响应的数据结构
llm_client.py         调用阿里 Qwen 模型
resume_parser.py      解析 PDF 和图片简历
routes/interview.py   处理面试追问和报告生成接口
routes/resume.py      处理简历上传解析接口
main.py               创建 FastAPI 应用
```

## 动态追问流程

系统不是一次性生成所有问题，而是逐题生成。

流程：

1. 用户填写简历、岗位 JD、公司要求和面试强度。
2. 前端请求后端生成第一题。
3. 用户回答第一题。
4. 前端把历史回答和下一阶段发给后端。
5. 后端调用模型生成下一题。
6. 循环直到面试结束。
7. 后端根据所有问答生成报告。

这个设计的好处是：后面的问题可以根据前面的回答变化，更像真实面试。

## 面试强度

当前支持三种模式：

- 快速体验：5 题。
- 标准训练：8 题。
- 深度模拟：12 题。

深度模拟会增加更多项目和技术追问，比如简历风险、项目难点、技术追问、系统设计、场景问题等。

## 简历解析

PDF 简历：

- 后端使用 `pypdf` 提取文本。
- 提取出的文本会作为简历摘要。

图片简历：

- 后端把图片转成 Base64。
- 调用视觉模型识别简历内容。
- 模型返回简历摘要。

MVP 阶段不保存用户上传的原始文件。

## 模型参数选择

问题生成：

- temperature 稍高。
- 目的是让面试官更自然、更有追问感。

报告生成：

- temperature 较低。
- 目的是让评分和反馈更稳定。

## 错误兜底

如果模型接口失败，前端会使用本地兜底逻辑。

例如：

- 生成问题失败时，用本地问题模板继续面试。
- 生成报告失败时，用本地估算报告。

这样用户不会因为模型接口偶尔失败就完全无法使用。

## 报告保存失败怎么处理

报告生成和报告保存是两个动作。

设计原则：

- 报告生成成功后，先展示给用户。
- 数据库保存成功时，页面显示“已保存到数据库”。
- 数据库保存失败时，前端把报告暂存到 localStorage。
- 页面显示“数据库保存失败，已本地暂存”。
- 用户可以点击“重新保存”再次尝试写入后端数据库。
- 后端保存异常时记录日志，方便排查数据库或字段问题。

这样可以保证用户体验不会被数据库异常直接打断。

## 当前不足

- 数据库目前是 SQLite，适合开发阶段。
- 还没有数据库迁移工具。
- 还没有用户登录。
- 简历解析还比较基础。
- 还没有接入 RAG。
- 还没有部署到公网服务器。

## 常见面试追问

### 为什么不让前端直接调用模型？

因为 API key 会暴露在浏览器中，不安全。后端可以保护密钥，并统一管理 prompt 和错误处理。

### 系统怎么决定下一题？

前端会把候选人档案、历史回答和下一阶段发给后端。后端把这些信息组织成 prompt，让模型生成下一题。

### 怎么保证模型输出稳定？

后端要求模型输出 JSON。生成报告时使用较低 temperature，让评分更稳定。

### 模型接口失败怎么办？

前端有本地兜底逻辑，可以继续生成问题和基础报告。

### 如果要加 RAG，你会怎么设计？

我会设计两个知识库：

- 岗位知识库：岗位 JD、题库、技术知识、评分标准。
- 候选人画像库：简历、历史回答、薄弱点、历史报告。

生成问题和报告时，后端先检索相关内容，再把检索结果放进 prompt。

当前项目已经实现了第一版轻量 RAG：

- 本地 JSON 岗位知识库。
- 关键词检索。
- 把检索结果放进动态追问和报告生成 prompt。
- 基于 SQLite 历史记录的候选人画像 RAG。
- 根据历史风险点和训练建议增强后续追问。

后续可以升级为 embedding + 向量数据库。
