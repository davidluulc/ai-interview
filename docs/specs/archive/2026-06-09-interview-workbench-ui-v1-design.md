# 面试训练工作台 UI 重构 V1 设计

## 1. 文档目的

本文档用于约束 AI 模拟面试系统下一轮前端体验增强。

阶段一“面试体验增强 V2”已经完成，后端已经具备学习辅导模式、真实面试模式、重复问题保护、Agent 决策日志、逐题复盘和训练计划。当前主要问题不再是“系统能不能问”，而是“用户在页面上是否感觉自己正在进行一场真实、清晰、可复盘的模拟面试”。

本轮目标是把当前面试区域从“表单 + 大卡片”升级为“面试训练工作台”。

## 2. 当前页面问题

根据当前页面截图和已有代码，主要问题包括：

- 面试进度使用 8 个大胶囊展示，占用首屏空间过多。
- 当前问题展示像普通卡片，不像真实对话。
- 对话历史、当前问题、回答输入区之间的层级不够清晰。
- Agent 决策信息虽然已经展示，但视觉上还不够像“追问依据”。
- 左侧投递档案面板信息密度偏高，容易抢占面试主区域注意力。
- 面试页面整体可用，但产品感还不够强。

这些问题会影响两件事：

- 用户自己用系统练面试时，沉浸感不足。
- 项目展示给面试官或同学看时，第一眼不够像成熟 AI 应用产品。

## 3. 本轮目标

本轮前端重构目标是：

```text
让用户进入面试后，第一眼看到的是一个清晰、紧凑、像真实 AI 面试训练产品的工作台。
```

具体目标：

- 把面试主体改成聊天流。
- 把进度展示改成轻量横向进度线。
- 把 Agent 决策展示成紧凑解释条。
- 把回答输入区改成更像面试回答输入框。
- 左侧投递档案保持可用，但减少视觉重量。
- 报告区域只做样式衔接，不在本轮重构核心范围内大改。

## 4. 页面布局设计

### 4.1 总体结构

页面仍保留左右结构：

```text
左侧：投递档案与面试设置
右侧：面试训练工作台
```

右侧工作台结构调整为：

```text
顶部状态栏
轻量进度线
聊天流区域
当前 Agent 解释条
回答输入区
操作按钮
```

推荐结构：

```text
Interview Workbench
├─ Workbench Header
│  ├─ 当前岗位 / 面试类型
│  ├─ Agent 模式
│  └─ 当前轮次
├─ Compact Progress
│  └─ 1 2 3 4 5 6 7 8
├─ Conversation Stream
│  ├─ AI 面试官消息
│  ├─ 用户回答消息
│  └─ AI 追问消息
├─ Agent Insight Bar
│  ├─ 考察点
│  ├─ 难度
│  ├─ 下一步动作
│  └─ 触发规则
└─ Answer Composer
   ├─ textarea
   ├─ 回答状态提示
   ├─ 提交回答
   └─ 生成报告
```

## 5. 组件设计

### 5.1 顶部状态栏

当前顶部的 `SESSION`、`1 / 8`、`真实模型` 保留，但做成更紧凑的信息栏。

应展示：

- 当前岗位或面试类型。
- 当前 Agent 模式：学习辅导模式 / 真实面试模式。
- 当前轮次：例如 `1 / 8`。
- 当前生成来源：真实模型 / 本地兜底。

视觉要求：

- 不使用过大的统计卡片。
- 信息横向排列。
- 移动端可以换行，但不能遮挡或挤压主聊天区。

### 5.2 轻量进度线

替换当前大胶囊式 `stageStepper`。

新进度线规则：

- 每一轮是一个小圆点或短标签。
- 当前轮高亮。
- 已完成轮次显示完成状态。
- 未开始轮次弱化。
- 不再显示过长的阶段标题。

阶段名称可以放到悬浮提示或 `aria-label` 中，不强制在首屏完整展示。

验收标准：

- 桌面端进度条高度明显低于当前版本。
- 8 轮、5 轮、12 轮都不能挤爆布局。
- 移动端可以横向滚动或自动换行，但不能导致文本重叠。

### 5.3 聊天流区域

当前 `conversationList` 改成面试对话流。

消息类型：

- AI 面试官消息。
- 用户回答消息。
- 系统提示或兜底提示。

每条消息应包含：

- 角色标识。
- 问题或回答内容。
- 可选的考察点。
- 可选的难度或 Agent 动作。

视觉要求：

- AI 面试官和用户消息要明显区分。
- 消息宽度不能铺满整行，避免读起来像文档段落。
- 长文本自动换行。
- 空回答显示“未作答”，但不要让页面显得异常。

### 5.4 Agent 解释条

当前 `agentDecisionPanel` 保留，但重构成 `Agent Insight Bar`。

展示内容：

- Agent 模式。
- 下一步动作：例如 `lower_difficulty`、`switch_topic`。
- 当前考察点。
- 难度。
- 触发规则。
- 决策摘要。

展示原则：

- 面向普通用户：文案要解释“为什么这么问”。
- 面向开发者：保留 action、triggerRules 等工程字段。
- 不做成大段日志，不抢聊天区主视觉。

示例：

```text
考察点：RAG 召回链路
动作：lower_difficulty
原因：上一轮回答偏弱，先拆小问题。
触发规则：weak_answer
```

### 5.5 回答输入区

当前 `answerInput` 改成更像聊天输入区的回答 composer。

要求：

- 输入框高度适中，不能首屏过高。
- placeholder 更偏面试表达，例如：

```text
请输入你的回答。可以先讲结论，再补项目例子和实现细节。
```

- 操作按钮右对齐。
- `提交回答，进入下一题` 是主按钮。
- `直接生成报告` 是次按钮。
- 空回答提示应靠近输入框展示。

### 5.6 左侧投递档案

左侧不做大重构，只做密度优化。

建议：

- 顶部说明压缩，不要占太多高度。
- 岗位匹配 Agent 卡片更紧凑。
- 简历上传入口保留。
- textarea 保持可用，但视觉上不要比面试主区域更重。

## 6. 响应式设计

桌面端：

- 保持左右布局。
- 左侧宽度控制在 360px 到 430px。
- 右侧聊天区优先获得空间。

平板和窄屏：

- 左右布局可变成上下布局。
- 面试工作台优先展示在前。
- 进度线可横向滚动。

移动端：

- 输入区、聊天区、Agent 解释条不能互相遮挡。
- 按钮不能出现文字溢出。
- 不使用复杂悬浮层作为核心交互。

## 7. 测试策略

本轮仍然遵循测试驱动。

前端测试优先覆盖：

- `frontend_interview_flow.test.mjs`
  - 能渲染 Agent 模式。
  - 能渲染紧凑 Agent 决策信息。
  - 能显示聊天流中的 AI 问题和用户回答。
  - 能显示下一轮训练计划入口或报告内容。

- 可新增 `frontend_workbench_layout.test.mjs`
  - 检查关键容器 class 是否存在。
  - 检查进度线不再依赖大胶囊文案。
  - 检查 Agent Insight Bar 包含模式、动作、触发规则。

浏览器验证：

- 使用 Codex 内置浏览器打开 `http://localhost:8000/`。
- 验证桌面视口页面不乱。
- 验证移动宽度下文本不重叠。
- 验证开始面试、提交回答、生成报告的主链路可用。

## 8. 非目标

本轮明确不做：

- 不引入 React、Vue、Next.js。
- 不重写后端 API。
- 不做 Docker、Nginx、云服务器上线。
- 不接入 LangGraph。
- 不重构数据库。
- 不新增管理员后台。
- 不大改报告生成逻辑。
- 不把全部页面重做成新产品。

## 9. 验收标准

本轮完成后应满足：

- 面试主体区域呈现为聊天流，而不是单个大问题卡片。
- 进度条比当前版本更轻量，不再占据大量首屏空间。
- 当前 Agent 模式、考察点、动作、触发规则能被清晰看到。
- 回答输入区更像面试回答 composer。
- 左侧档案区没有明显抢占主面试区注意力。
- 桌面端布局稳定，没有明显空白失衡。
- 移动端文本不重叠，按钮不溢出。
- 前端测试脚本通过。
- 后端测试不受影响。

## 10. 面试表达

可以这样介绍本轮前端重构：

> 我在项目里没有只关注后端 Agent 和 RAG，也做了面试训练工作台的前端体验重构。原来的页面更像表单和卡片堆叠，用户沉浸感不强。我把面试区改成聊天流，用轻量进度线展示轮次，用 Agent Insight Bar 展示当前考察点、难度、动作和触发规则。这样用户既能像真实面试一样对话，也能看到系统为什么这么追问，符合 AI 应用“可解释、可复盘”的产品思路。

```text

要求：

```

## 12. 后续衔接

本轮 UI 重构完成后，再进入阶段二：Agent 工程化增强。

阶段二将重点处理：

- Agent 节点拆分。
- 轻量状态机设计。
- Agent Trace。
- Tool Calling 设计。
- LangGraph 迁移预留。

UI 重构先完成，可以让后续 Agent Trace、节点日志、状态机流转有更清晰的展示位置。
