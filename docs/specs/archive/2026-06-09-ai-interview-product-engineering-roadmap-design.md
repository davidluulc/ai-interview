# AI 模拟面试系统产品化与工程化升级 Spec

## 1. 文档目的

本文档用于约束 AI 模拟面试系统后续升级路线，避免项目继续以“想到什么做什么”的方式堆功能。

当前项目已经具备用户系统、投递档案、简历解析、三类 RAG、RAG 文档管理、混合检索、rerank、RAG 日志、Agent V1.5、历史复盘和前端页面。下一阶段的重点不是继续盲目加功能，而是把这些能力整理成一条清晰的产品和工程化主线。

本文档回答四个问题：

- 当前项目处在什么阶段。
- 后续应该先补什么、后补什么。
- 每个阶段做到什么程度才算完成。
- 如何把项目整理成简历和面试中能讲清楚的项目。

## 2. 当前项目状态

面向大学生和求职者的 AI 模拟面试训练系统。

## 3. 当前核心问题

虽然当前系统功能已经较多，但仍存在以下问题：

### 3.1 面试体验仍需增强

当前系统已经支持动态追问，但真实体验仍可能出现：

- 问题和用户真实能力不匹配。
- 用户连续答不上来时，系统仍可能停留在同一知识点附近。
- 学习辅导模式和真实面试模式的差异还需要更明显。
- 报告需要进一步变成“训练方案”，而不是只给分数和建议。

### 3.2 Agent 工程化还停留在 V1.5

当前 Agent 已具备状态、决策、日志和兜底，但还不是完整工作流框架：

- 没有正式 Tool Calling 协议。
- Agent 节点还没有拆成独立 service。
- Agent 状态还没有持久化 checkpoint。
- 还没有 LangGraph StateGraph。
- human-in-the-loop 还停留在设计层面。

### 3.3 RAG 工程化需要更贴近真实业务

当前 RAG 已经具备 BM25、向量、混合检索和 rerank，但还需要补强：

- 知识库样例数据不足。
- Python 后端岗与 AI 应用开发岗的知识体系还需要持续扩充到 seed 数据和评估 case。
- metadata filter 还不够完整。
- 普通用户视图和调试视图需要进一步区分。
- RAG 质量评估需要从测试脚本走向可观察面板。
- 引用来源和“为什么命中这些资料”需要更清楚。

### 3.4 上线部署尚未开始

项目目前仍适合本地开发和学习展示，尚未进入生产部署阶段：

- 默认数据库仍是 SQLite。
- 未接入 PostgreSQL 生产环境。
- 未接入 Redis。
- 未编写 Docker Compose。
- 未配置 Nginx、HTTPS、云服务器部署。

## 4. 最终目标

项目最终目标不是做一个简单 AI 聊天页面，而是做成一个能体现 AI 应用开发能力的完整项目：

```text
RAG 提供依据，
Agent 控制流程，
后端保障稳定性，
前端提供训练体验，
日志和评估保证可观测，
部署链路证明工程落地能力。
```

最终希望达到的效果：

- 用户可以上传简历或填写投递档案。
- 用户可以输入岗位 JD 和公司要求。
- 系统结合岗位知识库、题库和候选人画像生成面试问题。
- Agent 根据回答质量、剩余轮次、RAG 命中质量动态调整策略。
- 用户可以选择学习辅导模式或真实面试模式。
- 面试结束后生成报告、逐题复盘和训练建议。
- 开发者可以查看 RAG 命中日志和 Agent 决策日志。
- 项目可以部署到云服务器，并能作为简历项目展示。

## 5. 升级路线总览

后续升级分四个阶段：

```text
阶段一：面试体验增强
阶段二：Agent 工程化增强
阶段三：RAG 工程化增强
阶段四：上线部署与产品化
```

推荐顺序是：

1. 先增强面试体验，因为这是用户最直接感受到的价值。
2. 再增强 Agent 工程化，因为这是 AI 应用开发岗最容易被深挖的亮点。
3. 再增强 RAG 工程化，因为 RAG 需要足够数据和评估样例支撑。
4. 最后做上线部署，因为部署前需要先保证产品主链路值得展示。

截至 2026-06-09 的当前进度：

| 阶段 | 状态 | 当前说明 |
| --- | --- | --- |
| 阶段一：面试体验增强 | 基本完成，后续继续体验打磨 | 已有学习辅导 / 真实面试模式、连续弱回答处理、重复问题保护、报告复盘和面试工作台 UI。 |
| 阶段二：Agent 工程化增强 | 基本完成第一轮 | 已有 Agent State、ToolCalls、nodeTrace、Agent 决策日志和 LangGraph 迁移预留设计。 |
| 阶段三：RAG 工程化增强 | 进行中 | 已有 RAG metadata、RAG 命中日志、RAG 评估基础；当前进入“知识体系与评估数据集 V2”，重点补 AI 应用开发岗和 Python 后端岗知识库样例。 |
| 阶段四：上线部署与产品化 | 未开始 | 仍未进入 Docker、Nginx、PostgreSQL、Redis、云服务器部署阶段。 |

知识库扩展原则：

```text
知识地图尽量全覆盖，seed 数据分批落地。
```

Python 后端岗和 AI 应用开发岗的完整知识地图会作为长期学习目录保留，并在后续迭代中陆续进入 RAG seed。但不会机械地把每个小知识点一次性塞进 seed。每个知识点进入 seed 前应满足至少一个条件：

- 高频面试会问。
- 能支撑当前 AI 模拟面试体验。
- 能被 RAG evaluation case 验证。
- 能形成清晰追问、评分点和风险信号。
- 能帮助用户解释本项目本身。

因此，第 5、6 节知识地图中的内容会持续扩充进 seed，但优先级按“高频、可追问、可评估、贴合项目”排序。

## 6. 阶段一：面试体验增强

### 6.1 阶段目标

让系统从“能模拟面试”升级为“能帮助用户训练面试”。

重点是：

- 问题更自然。
- 追问更有依据。
- 连续答不上来时不机械卡死。
- 学习辅导模式和真实面试模式有明显差异。
- 报告能指导下一轮训练。

### 6.2 主要能力

阶段一建议开发：

- 面试模式体验增强：
  - 学习辅导模式：更偏解释、降难度、给训练方向。
  - 真实面试模式：保留压力，但避免无效重复追问。
- 问题生成策略增强：
  - 每道题必须有考察点。
  - 每道题必须有追问依据。
  - 避免连续围绕同一个细节重复提问。
- 答不上来处理策略：
  - 第一次不会：降低难度。
  - 连续两次不会：拆成基础概念。
  - 连续三次不会：切换话题或进入学习提示。
- 报告增强：
  - 按题输出为什么问。
  - 当前回答缺少什么。
  - 下次应该怎么答。
  - 给出训练任务。
- 前端增强：
  - 当前 Agent 模式展示更明显。
  - 当前问题展示“考察点”和“追问依据”。
  - 报告展示“下一轮训练计划”。

### 6.3 验收标准

阶段一完成后应满足：

- 用户能明确选择学习辅导模式和真实面试模式。
- 连续答不上来时，系统不会一直问同一个细节。
- 每道问题能展示考察点或追问依据。
- 报告里有逐题复盘和训练动作。
- 至少有一轮完整面试可以明显体现两种模式差异。
- 后端测试覆盖连续弱回答、切换话题和报告复盘。
- 前端测试覆盖模式选择和 Agent 决策展示。

## 7. 阶段二：Agent 工程化增强

### 7.1 阶段目标

让 Agent 从“可解释决策层”升级为“可拆分、可追踪、可迁移的工作流系统”。

### 7.2 主要能力

阶段二建议开发：

- Agent service 模块化：
  - `observe_state`
  - `retrieve_context`
  - `analyze_answer`
  - `select_action`
  - `generate_question`
  - `update_memory`
- Tool Calling：
  - 把 RAG 检索、回答分析、记忆更新抽象为工具。
  - 记录每次工具输入和输出摘要。
- Agent Trace：
  - 记录节点执行顺序。
  - 记录每个节点的输入摘要和输出摘要。
  - 记录失败和 fallback。
- 状态机化演进：
  - 当前不直接引入完整状态机框架。
  - 先定义 `InterviewSessionState` 和事件。
  - 后续当暂停、恢复、跳题、人工确认稳定后，再抽象状态机。
- LangGraph 迁移：
  - 将 V1.5 节点映射到 LangGraph StateGraph。
  - 引入 checkpoint。
  - 引入 human-in-the-loop。

### 7.3 状态机演进方向

当前阶段不直接写状态机代码，但在设计上预留以下状态：

```text
idle
collecting_profile
ready
asking
waiting_answer
analyzing_answer
deciding_next_action
generating_question
generating_report
completed
failed
```

典型事件：

```text
START_INTERVIEW
QUESTION_GENERATED
ANSWER_SUBMITTED
ANSWER_ANALYZED
DECISION_SELECTED
REPORT_REQUESTED
REPORT_GENERATED
ERROR_OCCURRED
RESET_SESSION
```

面试流程稳定后，可以将：

```text
当前状态 + 事件 -> 下一个状态
```

抽象为 `InterviewSessionStateMachine`。

### 7.4 验收标准

阶段二完成后应满足：

- Agent 节点边界清晰。
- 每轮 Agent 决策有 trace。
- 工具调用能被日志观察。
- fallback 决策可追踪。
- 能讲清楚自研 Agent 和 LangGraph 的关系。
- 有 LangGraph 迁移 spec 或 demo。

## 8. 阶段三：RAG 工程化增强

### 8.1 阶段目标

让 RAG 从“检索增强能力”升级为“可管理、可评估、可解释的知识系统”。

### 8.2 主要能力

阶段三建议开发：

- 知识库样例数据建设：
  - AI 应用开发岗。
  - Python 后端岗。
  - RAG 开发岗。
  - Agent 开发岗。
- 岗位知识地图建设：
  - Python 后端岗从语言基础、FastAPI、数据库、鉴权、缓存、测试、部署逐步覆盖。
  - AI 应用开发岗从模型调用、Prompt、RAG、Agent、评估、可观测和前沿协议逐步覆盖。
  - 知识地图作为长期目录，seed 数据按批次落地。
- RAG seed 数据增强：
  - 每批新增知识必须有关键词、追问问题、评分点和风险信号。
  - 每批新增题库必须有参考答案、答题要点和标签。
  - 每批新增内容必须同步补评估 case。
- 文档解析增强：
  - PDF。
  - Markdown。
  - 文本。
  - 后续再考虑 Word 和 OCR。
- metadata filter：
  - 岗位标签。
  - 面试阶段。
  - 难度。
  - 来源。
  - 用户权限。
- 引用来源：
  - 问题生成时记录引用了哪些 chunk。
  - 报告中说明评分依据来自哪些知识点。
- RAG 质量评估：
  - Hit@K。
  - MRR。
  - 关键词覆盖率。
  - 空召回率。
  - 检索模式对比。
- 前端调试面板：
  - 普通用户看“依据和建议”。
  - 开发者看“命中分数、metadata、chunk、日志”。

### 8.3 验收标准

阶段三完成后应满足：

- 知识库数据足够支撑 AI 应用开发岗和 Python 后端岗模拟面试。
- Python 后端岗与 AI 应用开发岗都有稳定知识地图、seed 数据和评估 case。
- 检索结果能展示来源和 metadata。
- RAG 命中日志能解释每次召回。
- 质量评估能对比 BM25、vector、hybrid、rerank。
- 不出现跨用户数据泄露。

## 9. 阶段四：上线部署与产品化

### 9.1 阶段目标

让项目从本地学习项目升级为可公网访问的展示产品。

### 9.2 主要能力

阶段四建议开发：

- PostgreSQL：
  - 替换生产环境 SQLite。
  - Alembic 管理迁移。
- Redis：
  - token 黑名单。
  - 限流。
  - 会话缓存。
  - 后续任务状态缓存。
- Docker Compose：
  - FastAPI。
  - PostgreSQL。
  - Redis。
  - Nginx。
- Nginx：
  - 反向代理。
  - 静态资源。
  - HTTPS。
- 管理后台：
  - 用户管理。
  - 知识库管理。
  - 日志查看。
- 云服务器部署：
  - 阿里云 ECS。
  - 域名。
  - HTTPS。
  - 环境变量和密钥管理。

### 9.3 验收标准

阶段四完成后应满足：

- 项目能通过公网访问。
- 数据保存在 PostgreSQL。
- Docker Compose 可以一键启动主要服务。
- Nginx 能代理前端和后端 API。
- 敏感配置不提交到 Git。
- 有中文部署文档和故障排查文档。

## 10. 非目标

当前总路线明确以下非目标：

- 阶段一不做云服务器部署。
- 阶段一不重构 Vue / React。
- 阶段一不引入完整状态机框架。
- 阶段一不引入 LangGraph 依赖。
- 阶段一不做企业级管理员系统。
- 阶段一不做复杂多租户权限。
- 阶段一不做完整 OCR 流水线。

这些能力不是不做，而是放到后续阶段。

## 11. 文档拆分方式

后续每个阶段都应该拆成独立 plan：

```text
总路线 Spec
-> 阶段一 Plan：面试体验增强
-> 阶段二 Plan：Agent 工程化增强
-> 阶段三 Plan：RAG 工程化增强
-> 阶段四 Plan：上线部署与产品化
```

每份 plan 应包含：

- 目标。
- 需要改动的模块。
- 测试策略。
- 验收标准。
- 学习重点。
- 面试表达。

## 12. 面试表达建议

可以这样介绍项目路线：

> 我做的是一个面向大学生和求职者的 AI 模拟面试系统。系统不是简单聊天，而是由 RAG 提供岗位、题库和候选人画像依据，由 Agent 控制面试流程，根据回答质量和剩余轮次决定深挖、降难度、切换话题或结束。当前项目已经具备用户系统、投递档案、三类 RAG、RAG 日志、Agent 决策日志和面试复盘。后续路线分为四步：先提升面试体验，再增强 Agent 工程化，然后完善 RAG 评估和知识库管理，最后做 Docker、Nginx、PostgreSQL 和云服务器上线。

如果被问为什么现在不直接上 LangGraph 或状态机，可以这样说：

> 当前阶段我先用轻量 Agent State 和规则兜底保证流程可控。因为面试业务还在快速迭代，过早引入复杂框架会增加维护成本。等面试流程稳定后，我会把状态、事件和节点抽象出来，再迁移到 LangGraph StateGraph，并通过 checkpoint 支持暂停恢复和人工确认。
