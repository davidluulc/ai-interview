# 逐题学习复盘 V1 设计说明

## 背景

当前 AI 模拟面试系统已经具备注册登录、投递档案、简历入口、面试提问、RAG 命中日志、RAG 文档管理、历史面试复盘等能力。现有报告主要包含总分、优势、风险和行动建议，适合做一次整体评价，但对学习型用户还不够友好。

用户在真实使用中暴露出两个问题：

- 当某个知识点答不上来时，面试官可能连续围绕同一个点追问，体验偏机械。
- 面试结束后，用户只能看到整体复盘，不能逐题知道自己为什么卡住、缺了什么、下一次应该怎样回答。

本版本目标是把报告从“单次评分结果”升级为“逐题学习辅导”，帮助用户在练习后能真正补知识、补表达、补项目讲解能力。

## 目标

逐题学习复盘 V1 要解决以下问题：

- 每一道面试题都能看到明确的考察点。
- 用户能知道自己的回答状态：完整、模糊、不会、跑题。
- 系统能解释为什么问这道题，而不是只展示问题文本。
- 系统能指出当前回答缺了哪些关键内容。
- 系统能给出参考回答方向，帮助用户下次组织语言。
- 系统能给出一个可执行的训练建议。
- 历史面试记录中也能查看逐题复盘，方便长期训练。

## 非目标

本阶段暂不实现以下能力：

- 不做完整学习计划系统。
- 不做错题本、知识卡片、每日训练任务。
- 不引入新的 Agent 框架。
- 不新增向量数据库、reranker 或 RAG 评估框架。
- 不重构前端技术栈。
- 不改变现有面试流程的主要交互方式。

这些能力可以放到后续 Agent V1 和学习教练模块中继续演进。

## 用户视角

面试结束后，用户点击“生成报告”或在历史记录中点击“查看复盘”，页面除了总分、优势、风险、行动建议以外，还展示“逐题学习复盘”。

每道题展示以下内容：

- 考察点：这道题主要考什么能力。
- 回答状态：完整、模糊、不会、跑题。
- 为什么问：这道题和岗位、简历、JD、RAG 命中资料或上一轮回答之间的关系。
- 缺失点：用户回答中缺少的关键概念、项目细节或表达结构。
- 参考回答方向：下次可以怎样组织答案。
- 训练建议：下一步应该补什么、练什么。

页面语气要偏学习辅导，不要像审判式扣分。例如优先使用“建议补充”“可以这样组织”“下一次重点练”这类表达。

## 数据结构设计

报告 JSON 在保留现有字段的基础上新增 `questionReviews`。

```json
{
  "score": 72,
  "strengths": ["能说明项目目标"],
  "risks": ["RAG 工程化细节解释不足"],
  "actions": ["准备一段 RAG 检索链路讲解"],
  "questionReviews": [
    {
      "index": 1,
      "focus": "RAG 召回链路",
      "question": "请说明你的 RAG 命中日志记录了哪些字段？",
      "answerStatus": "模糊",
      "whyAsked": "该问题用于确认你是否理解项目中的 RAG 可观测性设计。",
      "missingPoints": ["日志字段含义", "命中结果结构", "质量评估用途"],
      "referenceDirection": "可以按 query、retriever、hits、quality、used_in_prompt 的顺序说明。",
      "trainingAction": "准备一条能被 json.loads 解析的 RAG 日志示例，并解释每个字段。"
    }
  ]
}
```

字段约束：

- `questionReviews` 必须是数组。
- `index` 从 1 开始，对应面试题顺序。
- `focus` 优先使用面试时保存的 `focus`，缺失时由模型根据问题推断。
- `answerStatus` 只能使用 `完整`、`模糊`、`不会`、`跑题` 四类。
- `missingPoints` 最多 3 条，每条不超过 30 个中文字符。
- `referenceDirection` 不要求给标准答案，但要给答题组织方向。
- `trainingAction` 必须是可执行动作，不写空泛建议。

## 后端设计

### 报告生成

更新报告生成 prompt，让模型输出包含 `questionReviews` 的 JSON。

后端 `/api/interview/report` 仍然返回原有字段，同时透传 `questionReviews`。如果模型没有返回逐题复盘，后端需要生成一个兜底复盘，避免前端空白。

兜底规则：

- `focus` 使用答案中的 `focus`，没有则使用 `stage` 或 `综合能力`。
- `answerStatus` 根据回答文本简单判断：
  - 空回答、`不会`、`不知道`、`写不出来` 归为 `不会`。
  - 回答很短且没有解释过程，归为 `模糊`。
  - 回答明显和题目无关，V1 暂不复杂判断，可归为 `模糊`。
  - 其他情况归为 `完整`。
- `whyAsked` 使用通用解释，说明这题用于确认对应考察点。
- `missingPoints` 给出“概念解释”“项目例子”“验证方式”等基础项。
- `referenceDirection` 引导用户按“背景、做法、原因、结果”回答。
- `trainingAction` 给出一条可执行训练建议。

### 历史记录

历史记录不需要新增数据库字段，因为当前报告已用 JSON 保存。只要保存报告时包含 `questionReviews`，历史复盘就可以读取并展示。

## 前端设计

### 当前报告页

在现有报告卡片下方新增“逐题学习复盘”区域。

每道题使用独立卡片展示：

- 标题：`第 N 题 · 考察点`
- 状态标签：完整 / 模糊 / 不会 / 跑题
- 原问题
- 我的回答
- 为什么问
- 缺失点
- 参考回答方向
- 下一步训练

### 历史复盘页

历史复盘页也展示同样的逐题学习复盘。若旧历史记录没有 `questionReviews`，前端可以退回到现有的问答列表展示，避免旧数据无法查看。

## 错误处理

- 模型输出不是合法 JSON 时，沿用现有 `call_model` 的解析和兜底机制。
- 模型缺少 `questionReviews` 时，后端构造兜底复盘。
- 某道题缺少回答时，状态标记为 `不会`，并提示用户先补基础概念再练表达。
- 前端渲染时对缺失字段使用默认文案，避免页面空白或报错。

## 测试策略

后端测试：

- 报告接口返回 `questionReviews`。
- 模型返回完整逐题复盘时，后端能保留该结构。
- 模型不返回 `questionReviews` 时，后端能生成兜底复盘。
- 空回答或“不会”类回答会被标记为 `不会`。

前端测试：

- 当前报告页能渲染逐题学习复盘。
- 历史复盘页能渲染逐题学习复盘。
- 旧历史记录没有 `questionReviews` 时仍能正常展示问答。

## 面试讲法

这一阶段可以这样解释：

> 我发现普通评分报告对学习帮助有限，所以把报告升级成逐题学习复盘。每道题不只给对错，而是拆成考察点、回答状态、追问原因、缺失点、参考回答方向和训练建议。后端通过结构化 JSON 约束模型输出，并提供兜底逻辑，避免模型字段缺失导致前端不可用。历史记录仍然复用 report_json 保存，这样不用频繁改表，也方便后续演进到学习教练 Agent。

## 验收标准

- 完成一次模拟面试后，报告页能看到逐题学习复盘。
- 每道题至少展示考察点、回答状态、为什么问、参考回答方向和训练建议。
- 用户空回答或明确表示不会时，状态能显示为 `不会`。
- 历史面试复盘能展示逐题学习复盘。
- 旧历史记录没有逐题复盘时，页面不报错。
- 后端和前端测试通过。
