# Vue3 用户工作台 V2：面试训练与档案模块产品化重构设计

## 1. 文档目的

本文档用于规划 AI 模拟面试系统下一阶段的 Vue3 用户端重构。

上一阶段已经完成 Vue3 管理员后台 V1，解决了管理员与普通用户的入口差异，并把账号管理、RAG 质量诊断、Agent 决策日志和系统配置接入后台。

但从产品主线看，本项目的核心用户不是管理员，而是：

- 大三大四准备实习和校招的学生。
- 正在准备跳槽或转岗的社会求职者。

所以下一阶段应该回到用户端，把“投递档案 -> 模拟面试 -> 复盘报告 -> 专项训练”这条主流程做得更像一个正式产品，而不是继续堆管理员后台能力。

本阶段目标：

```text
把 Vue3 用户端从“页面骨架”升级为“面试训练工作台 V2”。
```

核心表达目标：

```text
我把系统拆成用户端工作台和管理员后台两条线。用户端负责求职者的面试训练闭环，包括投递档案、AI 模拟面试、RAG/Agent 解释、复盘报告和专项训练；管理员后台只负责账号、RAG 质量、Agent 日志和系统配置等可观测能力。这样产品边界更清晰，也更符合真实企业系统的前后台分离设计。
```

## 2. 产品边界

### 2.1 用户工作台负责什么

用户工作台面向普通求职者，核心任务是让用户更高效地准备面试。

应该包含：

- 投递档案管理。
- 当前档案选择。
- 模拟面试入口。
- 面试模式选择。
- 面试聊天体验。
- Agent 决策摘要。
- RAG 检索上下文解释。
- 面试结束后的复盘入口。
- 基于弱点的训练入口。

用户工作台的页面应该回答：

```text
我现在准备哪个岗位？
我当前使用哪份简历/JD/公司信息？
AI 面试官为什么这样问？
我的薄弱点是什么？
下一步应该训练什么？
```

### 2.2 管理员后台负责什么

管理员后台不应该承载用户自己的面试训练流程。

管理员后台应该负责：

- 账号只读管理。
- 平台概览。
- RAG 质量诊断。
- RAG 文档概览。
- Agent 决策日志。
- 系统配置只读展示。

管理员后台可以看到“统计”和“质量”，但不应该把“用户自己操作的面试训练”和“用户自己维护的档案”当作主要功能放进去。

合理边界：

```text
用户端：我怎么训练。
管理员后台：系统运行得怎么样。
AI 调试台：RAG / Agent 为什么这样运行。
```

### 2.3 AI 调试台暂不进入本阶段

本阶段不直接做独立 AI 调试台。

后续可以单独设计：

```text
AI Debug Console V1：RAG 命中、Agent 决策链路与 LangGraph checkpoint 调试台
```

但本阶段只在用户工作台里保留轻量解释区，例如：

- Agent 为什么这样问。
- 当前问题用到了哪些 RAG 上下文。
- 当前面试模式是学习辅导还是真实面试。

## 3. 当前问题

### 3.1 面试页面仍偏骨架

当前 `frontend/src/pages/app/InterviewPage.vue` 已经有：

- 面试聊天区。
- Agent Decision 侧栏。
- RAG Context 侧栏。
- 调用 `useInterviewStore`。
- 读取 `profiles.currentProfileId`。

但仍存在问题：

- 页面没有明确展示当前档案。
- 用户不知道是否已经选择简历/JD。
- 面试模式固定为 `coach`，没有可见选择。
- 开始面试、继续回答、结束面试之间的状态不够清楚。
- Agent 决策摘要和 RAG 上下文还偏“开发调试信息”，不是面向求职者的解释。
- 中文文案存在编码显示异常，需要统一修正。

### 3.2 档案页面已经有基础能力但体验还薄

当前 `frontend/src/pages/app/ProfilesPage.vue` 已经有：

- 创建档案表单。
- 档案列表。
- 设置当前档案。
- 调用 `useProfilesStore`。

但仍存在问题：

- 新建档案和已有档案挤在同一屏，信息密度不够优雅。
- 当前档案状态不够突出。
- 简历、JD、公司信息缺少结构化预览。
- 用户不知道“选中档案后去哪里面试”。
- 没有明显的“用此档案开始面试”动作。
- 中文文案存在编码显示异常。

### 3.3 训练闭环没有形成强入口

系统后端已经有训练任务、weakTags、报告复盘等能力，但 Vue3 用户端还没有把它们串成一个完整闭环：

```text
完成面试 -> 看到复盘 -> 生成训练建议 -> 进入训练任务
```

本阶段可以先做用户端的入口和信息架构，不必一次性把所有训练细节做满。

### 4.1 页面目标

```text

```

```text

/vue/app/knowledge
```

但不把范围扩大到全站视觉大改。

### 4.2 功能目标

本阶段完成：

- 档案页产品化：
  - 当前档案突出展示。
  - 新建档案表单更清晰。
  - 档案列表支持选择当前档案。
  - 每个档案提供“开始面试”入口。
  - 文案修正为正常中文。

- 面试页产品化：
  - 显示当前档案摘要。
  - 未选择档案时给出明确引导。
  - 支持选择 `学习辅导模式 / 真实面试模式`。
  - 面试聊天区域更像工作台。
  - Agent 决策解释改成用户能理解的表达。
  - RAG 上下文解释改成“本题依据”或“参考资料”表达。
  - 文案修正为正常中文。

- 训练页产品化：
  - 展示训练任务入口。
  - 明确训练任务和面试复盘之间的关系。
  - 如果当前还没有训练任务，显示空状态和行动建议。

### 4.3 工程目标

本阶段继续遵循 Vue3 工程化拆分：

- 页面组件负责布局。
- store 负责状态。
- API client 负责请求。
- 可复用 UI 逻辑抽到组件。
- 先写 Vitest，再实现。

## 5. 信息架构设计

### 5.1 用户端导航

用户端导航继续保留：

```text
面试
档案
知识库
复盘
训练
后台（仅管理员可见）
```

但要明确：

- `面试` 是用户主工作台。
- `档案` 是准备资料入口。
- `训练` 是薄弱项提升入口。
- `后台` 不是用户主流程，只对管理员展示。

### 5.2 档案页结构

推荐结构：

```text
顶部：页面标题 + 说明

当前档案区：
- 档案名
- 目标岗位
- 公司
- 简历/JD 完整度提示
- 开始面试按钮

新建档案区：
- 档案名称
- 目标岗位
- 目标公司
- 岗位 JD
- 简历概况

档案列表区：
- 档案卡片
- 当前档案标识
- 设为当前
- 开始面试
```

如果没有档案：

```text
显示空状态：先创建一个投递档案，AI 面试官会结合简历、JD 和岗位要求生成问题。
```

### 5.3 面试页结构

推荐结构：

```text
顶部：当前档案摘要 + 模式切换

左侧主区：
- 面试聊天
- 当前问题
- 用户回答输入框
- 提交回答
- 结束面试 / 查看复盘入口

右侧解释区：
- 为什么这样问
- 当前参考资料
- 当前难度和阶段
- 下一步训练建议入口
```

未选择档案时：

```text
不直接让用户进入空面试。
显示引导：请先选择或创建投递档案。
提供按钮：去创建档案。
```

### 5.4 训练页结构

推荐结构：

```text
顶部：训练中心标题 + 当前训练来源说明

训练任务列表：
- weakTag
- 任务标题
- 来源面试记录
- 状态：未开始 / 进行中 / 已完成 / 已归档
- 开始训练按钮

空状态：
- 暂无训练任务
- 先完成一次模拟面试，系统会根据薄弱点生成训练建议
```

## 6. 交互设计

### 6.1 档案到面试的跳转

用户在档案页点击“开始面试”后：

```text
selectProfile(profile.id)
router.push("/vue/app/interview")
```

面试页根据 `profiles.currentProfile` 展示当前档案摘要。

### 6.2 面试模式选择

面试页提供分段控制：

```text
学习辅导模式 coach
真实面试模式 interview
```

默认建议：

```text
coach
```

原因：

- 当前用户还在学习和准备项目表达。
- coach 模式更适合解释、降难度和补基础。
- interview 模式可以作为压力训练。

模式说明不应该写成长篇说明，只需要短标签和当前状态提示。

### 6.3 RAG / Agent 解释

用户端不要直接展示过多调试字段。

推荐把技术字段翻译成产品表达：

```text
Agent Decision -> 为什么这样问
RAG Context -> 本题参考资料
retrievalQuality -> 资料匹配程度
fallbackUsed -> 使用兜底策略
```

管理员后台和未来 AI 调试台可以保留更技术化的 JSON / 日志字段。

### 6.4 空状态和错误状态

必须补齐：

- 没有档案。
- 档案加载中。
- 档案保存失败。
- 面试请求失败。
- 当前 token 失效。
- RAG 上下文为空。
- Agent 决策摘要为空。

错误信息要用中文，避免直接暴露英文异常堆栈。

## 8. 后端边界

本阶段原则：

```text
优先不改后端 API。
```

只有当前端无法展示必要信息时，才做最小补齐。

允许的后端小改：

- 为已有档案接口补只读字段。
- 为训练任务接口补当前用户自己的训练任务列表。
- 为面试响应补更清晰的 `decisionSummary` 或 `ragReasons` 字段。

不进入本阶段：

- 不重构 RAG。
- 不重构 Agent。
- 不改 LangGraph 主流程。
- 不做管理员后台写操作。
- 不改数据库结构，除非发现现有训练任务字段完全无法满足当前页面展示。

## 13. 面试表达

可以这样讲：

```text
我在 Vue3 重构阶段把系统拆成用户端工作台和管理员后台。用户端工作台主要服务求职者，包括投递档案、模拟面试、复盘和训练闭环；管理员后台主要服务系统观察，包括账号列表、RAG 质量、Agent 日志和系统配置。

这样拆分的原因是，面试和档案是普通用户的核心业务流程，不应该塞进管理员后台。管理员后台最多看统计和质量，不直接承载用户训练操作。这个设计能体现我对前后台分离、权限边界和 AI 应用可观测性的理解。
```
