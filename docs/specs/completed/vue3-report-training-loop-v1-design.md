# Vue3 面试报告与训练闭环 V1 设计文档

更新时间：2026-06-12

## 1. 阶段定位

本阶段目标是把 Vue3 用户端从“能进入面试、能看到训练入口”升级为“面试结束后能进入复盘，并从复盘自然生成下一步训练任务”的完整闭环。

当前项目已经完成：

- Vue3 登录 / 注册 / 用户工作台基础路由。
- Vue3 档案页、面试页、训练页的阶段性产品化。
- 后端已有面试历史、报告保存、训练任务、weakTag、候选人画像等能力。
- RAG、Agent、LangGraph、管理员后台、Docker/Nginx 等阶段性能力已经落地，不是本阶段主线。

本阶段不重新开发 RAG、Agent、LangGraph 或部署工程，而是把已有后端能力通过 Vue3 前端串成用户能理解、能使用、能复盘、能继续训练的产品路径。

## 2. 用户目标

目标用户是正在准备实习、校招、社招面试的大学生和求职者。

用户完成一次模拟面试后，最关心的不是“系统内部调用了多少 RAG 和 Agent”，而是：

- 我这场面试整体表现怎么样？
- 哪些问题答得不好？
- 为什么系统问了这些问题？
- 我的薄弱点是什么？
- 下一步应该练什么？
- 练完之后能不能再回到下一轮面试？

所以本阶段要让用户看到一条清晰路径：

```text
选择档案
-> 开始面试
-> 完成面试
-> 查看报告
-> 看到逐题复盘和薄弱点
-> 生成或进入专项训练任务
-> 完成训练
-> 回到下一轮面试
```

## 3. 本阶段范围

### 3.1 要做

1. Vue3 面试报告页
   - 展示面试总览。
   - 展示分数、等级、优势、风险点、改进建议。
   - 展示逐题复盘。
   - 展示 weakTags / 薄弱点。
   - 展示“为什么问这个问题”的 Agent/RAG 解释摘要。

2. Vue3 历史复盘页
   - 从历史记录进入某一场面试详情。
   - 支持按档案查看历史面试记录。
   - 支持从历史记录跳转到报告页。

3. Vue3 训练任务真实闭环
   - 训练页不只显示空状态或静态入口。
   - 调用后端训练任务接口读取真实任务。
   - 支持开始训练、完成训练、归档训练任务。
   - 从报告薄弱点跳转到对应训练任务。

4. 前端状态与 API 封装
   - 新增或完善 report/history/training 相关 API client。
   - 新增或完善 Pinia store。
   - 保持登录态、token refresh、错误提示一致。

5. 前端测试
   - 优先写或更新 Vue3 前端测试。
   - 覆盖报告页渲染、历史复盘跳转、训练任务状态变化。
   - 保证不破坏已有 13 个前端测试文件。

6. 学习文档
   - 新增一篇中文学习文档，解释“报告页、历史复盘、训练闭环在产品和工程上怎么串起来”。

### 3.2 不做

本阶段明确不做：

- 不重构 RAG 检索算法。
- 不重构 Agent 决策算法。
- 不引入新的 LangGraph 主流程。
- 不做 Docker、Nginx、VPS、HTTPS 上线。
- 不做管理员后台 V2。
- 不做复杂图表库。
- 不做支付、短信、邮箱验证码、第三方登录。
- 不重写后端 API，除非发现 Vue3 闭环必须补一个很小的兼容接口。

## 4. 页面结构设计

### 4.1 用户端导航

Vue3 用户端应保持清晰分区：

```text
档案
面试
历史
训练
后台管理（仅管理员可见）
```

其中：

- 档案页负责管理投递档案、简历摘要、岗位 JD。
- 面试页负责进行当前面试。
- 历史页负责查看过往面试记录。
- 报告页负责查看单场面试复盘详情。
- 训练页负责根据报告薄弱点进行专项训练。

报告页可以是独立路由，例如：

```text
/vue/app/reports/:recordId
```

历史页可以是独立路由，例如：

```text
/vue/app/history
```

训练页继续使用：

```text
/vue/app/training
```

### 4.2 报告页信息架构

报告页建议分为五块：

1. 报告头部
   - 面试岗位。
   - 使用档案。
   - 面试模式：学习辅导 / 真实面试。
   - 面试时间。
   - 总轮数。

2. 结果总览
   - 综合评分。
   - 表现等级。
   - 主要优势。
   - 主要风险。
   - 下一步建议。

3. 逐题复盘
   - 问题。
   - 用户回答。
   - 系统评价。
   - 命中的薄弱标签。
   - 更好的回答方向。

4. 提问依据
   - Agent 决策摘要。
   - RAG 命中摘要。
   - 当前问题和简历/JD/知识库之间的关系。

5. 训练入口
   - 根据 weakTags 推荐训练任务。
   - 如果任务已生成，显示任务列表。
   - 如果没有任务，显示“根据本报告生成训练任务”入口。

### 4.3 历史复盘页信息架构

历史页建议分为三块：

1. 筛选区
   - 当前档案筛选。
   - 面试岗位筛选。
   - 时间排序。

2. 历史列表
   - 面试岗位。
   - 面试时间。
   - 轮数。
   - 综合表现摘要。
   - weakTags 摘要。
   - 查看报告按钮。

3. 空状态
   - 如果没有历史记录，引导用户先去选择档案并开始面试。

### 4.4 训练页信息架构

训练页要从“静态说明页”升级为“真实任务页”：

1. 训练概览
   - 待完成数量。
   - 已完成数量。
   - 高频薄弱点。

2. 训练任务列表
   - 任务标题。
   - 来源报告。
   - weakTag。
   - 训练状态：pending / in_progress / completed / archived。
   - 推荐练习方式。

3. 训练动作
   - 开始训练。
   - 标记完成。
   - 归档。
   - 回到相关报告。

## 5. 数据流设计

### 5.1 面试结束后的数据流

```text
用户完成面试
-> 后端保存 interview record
-> 后端生成 report / weakTags / training task 候选信息
-> 前端跳转或提示查看报告
-> 用户进入报告页
-> 报告页读取历史记录和报告详情
-> 用户点击训练入口
-> 训练页读取或生成相关训练任务
```

### 5.2 历史页到报告页

```text
历史页加载当前用户面试记录
-> 用户点击某条记录
-> 进入 /vue/app/reports/:recordId
-> 报告页根据 recordId 拉取详情
```

### 5.3 报告页到训练页

```text
报告页展示 weakTags
-> 用户点击某个薄弱点训练
-> 前端携带 recordId / weakTag 跳转训练页
-> 训练页展示相关任务
```

## 6. 接口边界

本阶段优先复用已有接口。具体接口名称以当前代码为准，开发前需要先扫描：

- `frontend/src/api/`
- `frontend/src/stores/`
- `backend_python/routes/history.py`
- `backend_python/routes/training.py`
- `backend_python/routes/interview.py`

如果已有接口足够，则只改前端。

如果发现前端闭环缺少必要字段，允许做小范围后端兼容增强，但必须遵守：

- 不破坏旧接口。
- 不改现有请求路径语义。
- 新字段必须可选。
- 后端测试先行。

## 7. 前端状态设计

建议新增或完善三个 store：

1. history store
   - 当前历史列表。
   - 加载状态。
   - 当前筛选条件。
   - 错误信息。

2. report store
   - 当前报告详情。
   - 当前 recordId。
   - 报告加载状态。
   - 训练任务生成状态。

3. training store
   - 训练任务列表。
   - 按状态分组的任务。
   - 开始 / 完成 / 归档动作。
   - 当前 weakTag 筛选。

如果现有 store 已经覆盖部分能力，应优先扩展现有 store，不为了名字好看而重复建状态。

## 8. 视觉与交互原则

继续保持当前 Vue3 前端的极简、克制、偏苹果官网气质，但报告和训练页属于工具页面，不能做成纯展示页。

视觉原则：

- 大面积留白，但信息密度不能过低。
- 关键结论优先展示，技术日志放到解释区。
- 按钮只用于明确动作。
- 卡片用于单条报告、单条训练任务、单个复盘条目。
- 不做夸张渐变、不做装饰性大图、不做复杂动效。

交互原则：

- 用户从任何报告都能回到历史页。
- 用户从报告能进入训练任务。
- 用户从训练任务能回到来源报告。
- 没有数据时必须有明确引导，不显示空白页。
- 移动端不能横向溢出。

## 11. 面试表达目标

本阶段完成后，项目可以这样讲：

```text
我没有把 AI 模拟面试系统停留在“一问一答”的 demo 层面，而是做了面试训练闭环。
用户完成面试后，系统会保存历史记录并生成复盘报告，报告里包含整体评价、逐题分析、薄弱标签和提问依据。
这些薄弱标签会继续驱动训练任务，用户可以针对某个薄弱点开始训练、标记完成，再回到下一轮面试。
所以这个系统不是一次性问答工具，而是一个围绕档案、面试、复盘、训练持续迭代的 AI 求职训练平台。
```

如果面试官追问技术实现，可以补充：

```text
前端使用 Vue3 + Vite + TypeScript 拆分页面和状态管理。
报告页、历史页、训练页通过路由串联，API client 负责和 FastAPI 后端通信，Pinia store 管理列表、详情、加载状态和错误状态。
后端负责保存面试记录、生成 weakTags 和训练任务，前端负责把这些数据组织成用户能理解的复盘和训练路径。
```

## 12. 风险与边界

主要风险：

- 后端报告详情字段可能不够完整，需要小范围兼容增强。
- 旧前端和 Vue3 前端可能同时存在，开发时不要误改旧页面主线。
- 训练任务数据如果为空，页面必须提供合理空状态。
- 报告页如果暴露太多技术日志，会让普通用户看不懂。

处理策略：

- 先扫描现有接口，再决定是否补后端字段。
- 前端优先产品化展示，技术细节收进“提问依据”区域。
- 所有新增页面先写测试，再实现。
- 保持旧接口兼容，避免破坏当前可运行版本。
