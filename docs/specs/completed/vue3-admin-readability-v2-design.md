# Vue3 管理员后台可读性 V2 设计

## 1. 背景

当前 Vue3 管理员后台已经能展示账号、RAG 质量诊断、RAG 文档概览、Agent 决策日志和系统配置。

但页面里的部分信息仍然偏工程字段，例如 `empty_recall`、`role_knowledge`、`fallback`。这些字段对开发者有意义，但对刚进入后台的人不够直观，容易产生“看到了日志，但不知道怎么判断问题”的感受。

本轮目标不是新增后台能力，而是把已有只读数据翻译成更容易理解的诊断视图。

## 2. 本轮范围

只改 Vue3 管理员后台前端体验。

本轮修改：

- 优化 `RAG 质量诊断` 的解释、问题类型、建议动作。
- 优化 `RAG 文档概览` 的知识库类型、状态、可见性和风险提示。
- 优化 `Agent 决策日志` 的动作、阶段、原因和兜底说明。
- 增加对应前端测试。

本轮不做：

- 不改后端 API。
- 不新增数据库字段。
- 不改 RAG 检索算法。
- 不改 Agent 决策逻辑。
- 不引入 LangGraph。
- 不做 Docker、Nginx、云服务器上线。

## 3. 设计原则

### 3.1 字段翻译

后台页面不直接把技术字段丢给用户，而是做一层映射：

- `empty_recall` -> `空召回：没有找到资料`
- `weak_recall` -> `弱召回：找到了但相关性弱`
- `unused_in_prompt` -> `未进入 Prompt：检索到了但没进入模型上下文`
- `role_knowledge` -> `岗位知识库`
- `question_bank` -> `题库`
- `candidate_memory` -> `候选人画像`
- `enabled` -> `启用中`
- `disabled` -> `已禁用`
- `public` -> `公共资料`
- `private` -> `仅自己可用`

### 3.2 诊断说明

每个技术区块都要回答三个问题：

- 这个板块看什么？
- 这个指标异常说明什么？
- 管理员下一步可以怎么排查？

### 3.3 Agent 日志可读性

Agent 日志不只展示原始 action，而是展示成一条决策轨迹：

- 当前阶段是什么。
- Agent 判断了什么。
- 为什么这么判断。
- 下一步准备怎么做。
- 是否触发兜底规则。

## 4. 验收标准

- 管理员后台能看到 `RAG 质量诊断` 的解释文案。
- RAG issue type 使用中文诊断标签。
- RAG 文档的知识库、状态、可见性使用中文名称。
- 重复 chunk 数大于 0 时出现风险提示。
- Agent fallback 展示为 `兜底规则已启用`，而不是裸露的 `fallback`。
- 页面不出现 `undefined`。
- `frontend/src/pages/app/admin-page.test.ts` 覆盖以上行为。

## 5. 面试表达

我对管理员后台做过一轮可读性升级。原来后台只是把 RAG 日志和 Agent 日志展示出来，但很多字段是工程字段，比如 empty_recall、fallbackUsed。后来我在前端增加了一层展示映射，把这些字段翻译成“空召回”“兜底规则已启用”等中文诊断，并补充建议动作。

这样做的价值是提升 AI 应用的可观测性。管理员不只是看到系统出了问题，还能初步判断问题来自知识库缺失、召回质量弱、Prompt 拼接异常，还是 Agent 决策进入了兜底逻辑。
