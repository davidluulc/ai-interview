# 上线部署前综合补强 Spec

## 1. 文档目的

本文档用于约束 AI 模拟面试系统在真正进入 Docker、Nginx、云服务器上线之前的综合补强路线。

当前项目已经具备用户系统、投递档案、三类 RAG、RAG 文档管理、混合检索、rerank、RAG 日志、Agent V1.5、Agent 调试面板、历史面试记录和面试报告。下一阶段不能继续零散堆功能，而要围绕“可展示、可解释、可学习、可部署前准备”形成一条连续工程化主线。

本文档要解决四个问题：

- 睡前启动追求目标模式后，Codex 一整晚应该按什么顺序工作。
- 哪些内容必须完成，哪些内容是时间充足时的候补任务。
- 每个阶段完成后如何验证代码质量。
- 每个阶段必须给用户输出什么中文学习文档，便于第二天复盘和面试准备。

本文档设计的任务池应足够支撑 9-12 小时执行，但不要求为了凑满时间而强行扩展范围。任何阶段遇到高风险阻塞时，应优先保证代码质量和学习产出，而不是硬推进。

## 2. 当前项目状态

当前项目定位：

```text
面向大学生、应届生和社会求职者的 AI 模拟面试训练系统。
```

当前已具备能力：

- FastAPI 后端。
- SQLite 本地数据库。
- 用户注册、登录、refresh token、退出登录。
- 投递档案：简历、岗位、JD、公司要求。
- 简历解析入口。
- 岗位匹配 Agent。
- 三类 RAG：
  - 岗位知识库 RAG。
  - 题库 RAG。
  - 候选人画像 RAG。
- RAG 文档管理。
- BM25、向量检索、hybrid search、rerank。
- RAG 命中日志和基础质量评估。
- Interview Orchestrator Agent：
  - Agent State。
  - Tool Calls。
  - Agent Decision。
  - coach / interview 双模式。
  - 连续弱回答识别。
  - 重复问题保护。
  - topic shift。
  - nodeTrace。
  - Agent 决策日志。
- 前端面试训练工作台。
- Agent 调试面板。
- 历史面试复盘和报告。

当前主要短板：

- RAG 数据样例和评估样例仍不够体系化。
- RAG 质量评估还没有形成稳定的“数据集 -> 指标 -> 面板 -> 学习文档”闭环。
- Agent 工程化已经有雏形，但还需要更清晰的状态机、工具抽象和可讲解文档。
- 面试报告还可以继续向“下一轮训练计划”升级。
- 后端工程化文档还不够系统，用户在面试时讲 FastAPI、数据库、日志、部署链路仍然容易断片。
- 真实上线部署尚未开始，部署知识需要先通过文档和配置清单准备好。

## 3. 总目标

本阶段总目标：

```text
在不进入真实云服务器上线的前提下，
把 AI 模拟面试系统补强为一个更适合写进简历、用于学习、能被面试深挖的工程化项目。
```

更具体地说：

- RAG 不只是能召回，还要能评估召回质量、解释命中原因、形成评估样例。
- Agent 不只是能生成下一题，还要能说明状态、工具、规则、兜底和后续 LangGraph 迁移方向。
- 报告不只是给分，还要能指导用户下一轮怎么练。
- 后端不只是能跑，还要能讲清模块边界、数据库关系、日志、错误处理和配置管理。
- 上线前不急着部署，但要让用户知道 Docker、Nginx、Redis、数据库、环境变量下一步怎么接。

## 4. 非目标

本阶段明确不做：

- 不做 Docker / Nginx / 云服务器真实上线。
- 不直接引入 LangGraph。
- 不安装 LangChain。
- 不重构为 React、Vue、Next.js。
- 不开发管理员后台。
- 不做支付、短信、邮箱验证码等商业化功能。
- 不大规模重写数据库结构。
- 不做无关目录清理和大重构。
- 不为了追求任务量而牺牲测试和可读性。

如需涉及以上内容，只能写成“上线前准备文档”或“后续升级方向”，不能直接改代码落地。

## 5. 执行策略

本阶段采用“总控 spec + 分阶段 plan + 阶段验收 + 学习文档”的方式执行。

执行原则：

- 每个阶段必须有明确输入、输出和验收标准。
- 每个阶段优先写测试或验收脚本，再实现。
- 每个阶段结束前必须运行相关测试。
- 每个阶段结束后必须写中文学习文档。
- 每个阶段结束后必须更新进度记录。
- 测试失败时，不进入下一阶段。
- 如果某个功能超过当前阶段范围，转为文档设计，不强行实现。

推荐节奏：

```text
阶段开始 -> 读 spec -> 写/更新 plan -> 写测试 -> 实现 -> 跑测试 -> 写学习文档 -> 更新进度 -> 进入下一阶段
```

## 6. 时间规划与任务池

本文档的任务池按 9-12 小时设计。

| 时间段 | 阶段 | 目标 | 主要产出 |
| --- | --- | --- | --- |
| 0-1 小时 | 阶段 0 | 项目体检与执行基线 | 架构复盘、测试基线、进度记录 |
| 1-4 小时 | 阶段 1A | RAG seed 与 evaluation case | 知识样例、评估样例、测试 |
| 4-6 小时 | 阶段 1B | RAG 指标与命中解释 | Hit@K、MRR、关键词覆盖率、命中解释 |
| 6-7.5 小时 | 阶段 1C | RAG 可观测前端增强 | RAG 质量面板或命中解释展示 |
| 7.5-9 小时 | 阶段 2A | Agent V3 设计与第一步 | 状态机/工具抽象 spec、第一批测试 |
| 9-10.5 小时 | 阶段 3 | 面试训练闭环增强 | 报告训练计划、复练题、下一轮推荐、学习文档 |
| 10.5-12 小时 | 阶段 4 | 上线前工程化准备文档 | 部署前检查清单、Docker/Nginx/Redis/数据库学习文档 |

执行优先级：

```text
必须完成：阶段 0
尽量完成：阶段 1A、1B、1C
时间允许：阶段 2A
超过 9 小时仍有余力：先启动阶段 3，再启动阶段 4
```

## 7. 阶段 0：项目体检与执行基线

### 7.1 目标

确认当前项目在继续开发前处于可理解、可测试、可恢复的状态。

### 7.2 工作内容

- 扫描项目目录，确认核心文件和文档位置。
- 阅读现有总路线文档、RAG 文档、Agent 文档、前端 UI 文档。
- 运行测试基线：
  - `python -m pytest -q`
  - 所有前端 `.mjs` 测试。
- 检查当前服务是否能启动。
- 记录当前已完成能力、进行中能力、未开始能力。
- 识别本轮不应触碰的高风险区域。

### 7.3 产出

- `docs/learning/00-项目当前架构与进度复盘.md`
- 更新总路线进度文档。
- 如发现风险，新增“问题记录”小节。

### 7.4 验收标准

- 能说清当前项目有哪些模块。
- 能说清本轮为什么优先做 RAG 工程化。
- 测试基线结果被记录。
- 没有开始大规模代码改动。

## 8. 阶段 1：RAG 工程化 V2

阶段 1 是本轮主战场。

目标：

```text
让三个 RAG 从“能检索”升级为“可评估、可解释、可展示质量”。
```

### 8.1 阶段 1A：RAG Seed 与 Evaluation Case

#### 工作内容

- 梳理当前 RAG seed 数据结构。
- 补充高价值知识样例，优先覆盖：
  - Python 后端岗。
  - AI 应用开发岗。
  - 本项目自身可讲解内容。
- 补充 evaluation case：
  - query。
  - 期待命中的知识库。
  - 期待命中的关键词。
  - 期待召回的 chunk 标识。
  - 最低关键词覆盖率。
- 避免一次性塞入所有知识点，只加入能被测试验证的样例。

#### 产出

- RAG seed 数据更新。
- RAG evaluation case 更新。
- 对应后端测试。
- `docs/learning/01-RAG知识库样例与评估数据集.md`

#### 验收标准

- 至少覆盖 Python 后端岗和 AI 应用开发岗的高频样例。
- 每条样例能对应至少一个可验证 evaluation case。
- 测试能证明新增样例被正确检索或评估。

### 8.2 阶段 1B：RAG 指标与命中解释

#### 工作内容

- 梳理现有 Hit@K、MRR、关键词覆盖率实现。
- 如果已有实现，补测试和文档。
- 如果实现不足，补最小可用版本。
- 为命中结果生成解释摘要：
  - 命中了哪个知识库。
  - 命中了哪些 chunk。
  - 为什么认为这条命中有价值。
  - 命中质量是 good、weak 还是 miss。
- 确保解释逻辑不依赖大模型也能工作，优先用规则和元数据生成。

#### 产出

- RAG 指标测试。
- RAG 命中解释函数或增强。
- `docs/learning/02-RAG质量评估指标HitK-MRR-关键词覆盖率.md`

#### 验收标准

- 能用测试说明 Hit@K 是什么。
- 能用测试说明 MRR 是什么。
- 能用测试说明关键词覆盖率是怎么计算的。
- 能从一次检索结果里看到命中解释。

### 8.3 阶段 1C：RAG 可观测前端增强

#### 工作内容

- 在不大改前端架构的前提下，增强 RAG 调试视图。
- 优先展示：
  - query。
  - 检索器名称。
  - 命中数量。
  - top chunk。
  - quality。
  - 关键词覆盖率。
  - 命中解释。
- 普通用户视图和调试视图要区分，不把页面变成日志墙。
- 移动端不出现横向溢出。

#### 产出

- 前端 `.mjs` 测试。
- `app.js` / `styles.css` 的小范围增强。
- 浏览器验证记录。
- `docs/learning/03-RAG可观测面板怎么设计.md`

#### 验收标准

- 用户能看到“为什么这些资料被召回”。
- 开发者能看到 RAG 质量摘要。
- 缺字段时不出现 `undefined`。
- 前端测试通过。

## 9. 阶段 2：Agent 工程化 V3

阶段 2 是时间允许时启动的第二主线。

目标：

```text
把现有 Agent V1.5 整理成更接近企业 AI 应用项目的 Orchestrator 结构。
```

### 9.1 工作内容

- 梳理现有 Agent State、Tool Calls、Decision、Log。
- 定义状态机雏形，但不直接引入 LangGraph。
- 把 Agent 节点进一步文档化：
  - observe_state。
  - analyze_answer。
  - retrieve_context。
  - select_action。
  - generate_question。
  - update_memory。
- 抽象工具调用边界：
  - 三个 RAG 是工具。
  - 问题生成是模型能力。
  - 日志记录是可观测能力。
- 增强 Agent 学习文档，而不是盲目重写代码。

### 9.2 产出

- `docs/superpowers/specs/2026-06-10-agent-engineering-v3-design.md`
- 如时间允许，生成对应 plan。
- 如时间允许，补第一批 Agent V3 测试。
- `docs/learning/04-Agent状态决策工具日志完整讲解.md`

### 9.3 验收标准

- 能讲清 Agent 和普通 LLM 调用的区别。
- 能讲清 Agent State 从哪里来。
- 能讲清 Tool Calls 如何支撑决策。
- 能讲清 fallback、normalize、guardrail 的价值。
- 能讲清未来迁移 LangGraph 时哪些节点可以映射成 graph node。

## 10. 阶段 3：面试训练闭环增强

阶段 3 是候补执行阶段。

目标：

```text
把面试报告进一步升级成下一轮训练计划。
```

工作内容：

- 复盘报告按题解释：
  - 为什么问。
  - 回答缺什么。
  - 下次怎么答。
  - 推荐复练题。
- 下一轮训练建议：
  - 优先复练知识点。
  - 推荐模式：coach 或 interview。
  - 推荐难度。
- 前端展示训练闭环摘要。

产出：

- 报告相关测试。
- 前端报告展示增强。
- `docs/learning/05-AI模拟面试训练闭环怎么讲.md`

验收标准：

- 报告不只是评分，而是能指导下一轮训练。
- 用户能知道自己下一步该练什么。

## 12. 质量闸门

每个代码阶段结束前必须执行：

```powershell
python -m pytest -q
```

如果改了前端，还必须执行：

```powershell
Get-ChildItem tests -Filter "*.mjs" | ForEach-Object { node $_.FullName; if ($LASTEXITCODE -ne 0) { exit $LASTEXITCODE } }
```

如果改了页面体验，还应使用内置浏览器验证：

```text
http://127.0.0.1:8000/
```

浏览器验证至少检查：

- 页面能打开。
- 无明显横向溢出。
- 无 `undefined`。
- 控制台无 error。
- 新增面板或页面区域存在。

文档阶段结束前必须检查：

- 是否有空泛表达。
- 是否有未填写的占位符。
- 是否能被用户第二天直接阅读学习。
- 是否包含“面试时怎么讲”。

## 13. 安全停止条件

出现以下情况时，停止继续扩展功能：

- 同一个测试失败连续三次仍无法定位。
- 某阶段需要大规模数据库迁移，但当前 spec 没有覆盖。
- 某阶段需要引入新框架或新中间件。
- 前端改动开始演变成全站重构。
- 后端接口兼容性有破坏风险。
- 任务开始偏离 RAG、Agent、训练闭环、后端工程化主线。

停止后应做：

- 写问题记录。
- 保留当前可运行状态。
- 不进入下一阶段。
- 在最终总结里说明卡点、已验证内容和建议下一步。

- 通过测试验证的代码改动。
- 明确说明昨晚做了什么、为什么做、怎么验收。
- 明确说明哪些内容没做、为什么没做。
- 一段可以用于面试表达的项目讲解。

- 阶段 0 完成。
- 阶段 1A、1B、1C 基本完成。
- 阶段 2A 至少完成设计文档或第一批测试。
- `docs/learning/` 下新增多篇学习文档。
